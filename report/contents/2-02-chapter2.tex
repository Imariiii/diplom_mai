\section{ПРОЕКТИРОВАНИЕ СИСТЕМЫ}

\subsection{Архитектура системы}

Разрабатываемая система построена по многоуровневой клиент-серверной архитектуре с разделением на серверную и клиентскую части. Серверная часть реализована на языке Python с использованием фреймворка FastAPI и обеспечивает генерацию нагрузки, сбор метрик, управление подключениями к тестируемым СУБД, хранение истории тестов, а также управление профилями схем и группами баз данных. Клиентская часть реализована на TypeScript с использованием фреймворка Next.js и предоставляет веб-интерфейс для управления тестированием, визуализации метрик в реальном времени и анализа результатов.

Взаимодействие между клиентской и серверной частями осуществляется по двум каналам: REST API~--- для CRUD-операций, запуска тестов и получения результатов; WebSocket~--- для трансляции метрик в реальном времени без периодического опроса сервера.

В процессе работы система взаимодействует с СУБД двух принципиально разных типов. Собственная СУБД системы~--- PostgreSQL~15, хранящая базу данных \texttt{project\_data},~--- предназначена для хранения истории тестов, наборов сценариев нагрузки, конфигураций подключений и результатов анализа. Вторая группа~--- целевые базы данных, работающие под управлением PostgreSQL, MySQL или MariaDB, над которыми непосредственно проводятся нагрузочные тесты. Такое структурное разделение позволяет изолировать служебные данные от тестируемых сред и исключить искажение результатов из-за операций сохранения метрик в ходе прогона.

Архитектура системы включает восемь серверных компонентов и клиентскую подсистему визуализации. Центральным серверным компонентом является модуль генерации нагрузки~--- ядро системы, реализующее параллельное выполнение SQL-запросов с заданным числом виртуальных пользователей, сбор метрик производительности и управление жизненным циклом теста. Модуль управления индексами сопровождает жизненный цикл индексов схемы в ходе прогона. Модуль управления подключениями предоставляет абстракцию подключения к различным СУБД, управление пулами соединений и проверку доступности. Слой диалектов СУБД инкапсулирует сбор внутренних метрик и SQL-специфику PostgreSQL, MySQL и MariaDB. Модуль управления состоянием БД обеспечивает снимок и восстановление данных при write-сценариях и верификацию целостности. Модуль генерации сценариев вместе со слоем профилей схем и групп баз данных формирует каталог нагрузки и привязку подключений к схемам. Модуль хранения содержит репозитории для работы с базой данных истории, реализующие операции создания, чтения, обновления и удаления записей о тестах, результатах, сценариях и профилях. Модуль сравнительного анализа представляет собой сервис для сопоставления результатов нескольких тестовых прогонов, вычисления описательной статистики, применения статистических критериев и формирования аналитических отчётов. Подсистема визуализации реализована на стороне клиента в виде компонентов Next.js и отвечает за отображение временных рядов метрик, сравнительных диаграмм и аналитических отчётов на основе данных, получаемых от сервера.

Каждый компонент реализован в виде отдельного модуля с чётко определённым интерфейсом взаимодействия, что обеспечивает возможность независимой модификации и тестирования компонентов. Архитектура системы допускает расширение перечня поддерживаемых СУБД путём добавления новых драйверов подключения без модификации остальной кодовой базы.

Взаимодействие компонентов системы представлено на рисунке~\ref{fig:architecture}. Клиентский браузер отправляет HTTP-запрос на запуск теста, сервер создаёт запись в базе данных истории и запускает фоновую задачу. Фоновая задача вызывает \texttt{LoadTester}, который через \texttt{DatabaseConnection} подключается к тестируемым СУБД, выполняет SQL-запросы и собирает метрики. Метрики передаются через callback-механизм: класс \texttt{TestStreamingCallback} получает агрегаты от \texttt{LoadTester}, через \texttt{ConnectionManager} рассылает их подписчикам по WebSocket и сохраняет в базе данных истории через репозитории. Слой групп баз данных и профилей схем используется на этапе конфигурации: при выборе группы баз данных система автоматически разрешает привязанный профиль схемы и набор сценария нагрузки.

\subsection{Модуль генерации нагрузки и сбора метрик}

При проектировании модуля генерации нагрузки определялись требования к параллельному выполнению SQL-запросов, точному замеру времени отклика, сбору системных и внутренних метрик СУБД, управлению индексами сценария, а также автоматическому управлению состоянием тестируемых баз данных.

Генерация нагрузки осуществляется с использованием асинхронной модели выполнения на базе библиотеки asyncio. Каждый виртуальный пользователь реализуется как асинхронная задача через \texttt{asyncio.Task}, выполняющая заданное число итераций SQL-запросов. Параллельное выполнение задач обеспечивается функцией \texttt{asyncio.gather()}, которая запускает все задачи одновременно и ожидает их завершения. Такой подход позволяет эффективно использовать ресурсы сервера без создания избыточного числа потоков операционной системы.

Для каждой единицы нагрузки~--- одиночного SQL-запроса или пользовательской транзакции~--- фиксируется время отклика с использованием функции \texttt{time.perf\_counter()}, обеспечивающей высокую точность измерений. Режим нагрузки задаётся полем \texttt{workload\_mode} набора сценария: значение \texttt{query} соответствует одиночным операциям, \texttt{transaction}~--- атомарным транзакционным последовательностям. Результаты накапливаются в структуре данных, из которой по завершении измерительной фазы вычисляются итоговые метрики времени отклика: среднее, медиана (p50), процентили p95 и p99, минимум, максимум, стандартное отклонение, а также показатели ошибок.

Сбор метрик производительности встроен в цикл работы виртуальных пользователей: при достижении интервала в одну секунду на каждом подключении агрегируются результаты за прошедшее окно и выполняются следующие действия:

\begin{enumerate}[label=\arabic*)]
    \item вычисляются агрегированные метрики за интервал на основе накопленных результатов запросов или транзакций, включая успешную и полную пропускную способность;
    \item через библиотеку \texttt{psutil} собираются системные метрики хоста;
    \item через слой диалектов СУБД собираются внутренние метрики движка;
    \item агрегат передаётся через callback-механизм для трансляции в реальном времени и сохранения в базе данных истории.
\end{enumerate}

Параллельно агрегаты персистируются в сущностях \texttt{TimeSeries} и \texttt{MetricSample}; назначение сущностей раскрыто в подразделе о модели данных.

\subsubsection{Перечень собираемых метрик}

Состав фиксируемых показателей организован в четыре логические группы. Единица измерения времени отклика и основной пропускной способности определяется полем \texttt{workload\_mode} набора сценария нагрузки: при значении \texttt{query} учитывается одиночная SQL-операция, при \texttt{transaction}~--- атомарная пользовательская транзакция, включающая все SQL-шаги и завершение фиксацией или откатом. От этого же параметра зависят подписи метрик в итоговом отчёте прогона и в интерфейсе.

Первая группа охватывает время отклика в миллисекундах за измерительную фазу прогона. По завершении нагрузки вычисляются среднее значение, медиана p50, процентили p95 и p99, минимум, максимум и стандартное отклонение. В транзакционном режиме замер охватывает полный цикл выполнения транзакции, а не отдельные инструкции внутри неё.

Вторая группа описывает пропускную способность. Успешная пропускная способность в поле \texttt{throughput} выражает число успешно завершённых единиц нагрузки в секунду~--- запросов в секунду при сценариях с одиночными операциями или транзакций в секунду при транзакционной нагрузке. Полная интенсивность в поле \texttt{attempt\_rate} фиксирует все завершённые единицы в секунду, включая завершившиеся с ошибкой. Для транзакционных сценариев дополнительно сохраняется вторичная интенсивность SQL-шагов в поле \texttt{qps}, а также число успешных и неуспешных транзакций, откатов, суммарное число SQL-инструкций и среднее число инструкций на одну транзакцию.

Третья группа включает системные метрики хоста, собираемые с интервалом в одну секунду через библиотеку \texttt{psutil}. Фиксируются загрузка центрального процессора в процентах, использование оперативной памяти в процентах и мегабайтах, интенсивность дисковых операций, скорости чтения и записи на диск в мебибайтах в секунду, а также входящий и исходящий сетевой трафик в тех же единицах. Итоговый снимок системных показателей сохраняется вместе с агрегированными результатами прогона.

Четвёртая группа объединяет внутренние метрики СУБД, получаемые через слой диалектов. За измерительную фазу накапливается изменение коэффициента попаданий в кэш, фиксируются размер буферного пула в мегабайтах с указанием соответствующего параметра конфигурации движка, число ожиданий блокировок и число взаимных блокировок, а также объёмы таблиц и базы данных в целом. Дополнительно учитываются частота ошибок и число активных соединений.

Метрики фазы прогрева маркируются отдельно и не передаются в модуль сравнительного анализа. Секундные агрегаты записываются в сущность \texttt{TimeSeries} и используются для построения графиков в реальном времени; индивидуальные замеры задержки накапливаются в \texttt{MetricSample} и служат основой для статистических критериев при попарном сравнении прогонов.

\subsubsection{Модуль управления индексами}

Одним из проектных решений является управление индексами схемы в рамках жизненного цикла тестового прогона. До начала основного этапа тестирования модуль создаёт индексы базы данных в соответствии со спецификацией выбранного набора сценария нагрузки; после его завершения индексы удаляются, восстанавливая исходное состояние схемы. Такой подход позволяет оценивать влияние индексирования на производительность в рамках управляемого эксперимента: один и тот же набор SQL-запросов выполняется на схеме с заданной конфигурацией индексов, а результаты прогонов с разными конфигурациями становятся сопоставимыми. Каждый индекс в спецификации описывается набором атрибутов: имя таблицы, перечень столбцов, тип, признак уникальности и необязательное условие для частичных индексов.

\subsubsection{Алгоритм сохранения и восстановления состояния базы данных}

Тесты с операциями записи изменяют данные в тестируемой базе данных, что исключает повторяемость прогонов без предварительного сброса состояния. Для решения этой проблемы спроектирован модуль управления состоянием базы данных. Перед тестом анализируются SQL-запросы сценария: при наличии write-операций фиксируется снимок состояния базы данных, а после завершения теста данные восстанавливаются до исходного состояния. Для предотвращения конфликтов при параллельном тестировании нескольких баз данных одного типа предусмотрена блокировка на уровне типа СУБД, гарантирующая последовательное выполнение операций сохранения и восстановления.

В системе предусмотрены две взаимозаменяемые стратегии управления состоянием, выбор между которыми задаётся конфигурацией. Нативная стратегия использует штатные CLI-утилиты конкретной СУБД для создания дампа и его последующего восстановления --- этот подход обеспечивает максимальную точность и скорость операций. SQL-стратегия создаёт внутрибазовые копии таблиц средствами самой СУБД и не требует наличия внешних утилит, однако временно увеличивает объём данных в тестируемой базе. Если CLI-утилиты недоступны, система автоматически переключается с нативной стратегии на SQL-стратегию. После восстановления модуль верификации сравнивает число строк и контрольные суммы таблиц с зафиксированным до теста снимком и формирует отчёт о расхождениях при их обнаружении.

\subsubsection{Механизм параметризации запросов}

Для моделирования реалистичной нагрузки реализован механизм параметризации SQL-запросов. Каждый параметр запроса в наборе сценария нагрузки может быть сгенерирован одним из следующих способов:

\begin{enumerate}[label=\arabic*)]
    \item \texttt{random\_int} --- случайное целое число из заданного диапазона от min до max;
    \item \texttt{random\_from\_table} --- случайное значение из указанного столбца существующей таблицы базы данных;
    \item \texttt{sequential\_int} --- последовательное целое число, инкрементируемое на каждой итерации;
    \item \texttt{uuid} --- случайно сгенерированный UUID версии 4;
    \item \texttt{fixed} --- фиксированное значение, заданное в конфигурации параметра;
    \item \texttt{random\_string} --- случайная строка заданной длины из алфавитно-цифровых символов;
    \item \texttt{random\_date} --- случайная дата из заданного диапазона.
\end{enumerate}

Такой подход позволяет варьировать данные в запросах и избегать эффектов кэширования планов выполнения, которые могут искажать результаты тестирования. Значения типа \texttt{random\_from\_table} кэшируются на уровне ключа \texttt{db\_key:table:column} для снижения числа дополнительных запросов к БД в ходе теста.

Перед основным этапом тестирования выполняется фаза прогрева с линейным наращиванием нагрузки и удержанием на заданном уровне для инициализации кэшей и буферов СУБД. Метрики прогрева не входят в итоговый анализ~--- учитывается только измерительная фаза, что обеспечивает сопоставимость прогонов.

\subsection{Проектирование слоя сценариев и групп баз данных}

Для обеспечения гибкого сопоставления нагрузочных сценариев с конкретными базами данных в системе спроектирован трёхуровневый слой управления сценариями. Данная архитектура позволяет описать логику нагрузки независимо от конкретной СУБД и автоматически подбирать конкретные SQL-запросы при выборе подключения.

Нижний уровень образует шаблон сценария~--- логическое описание вида нагрузки без привязки к конкретной схеме данных. Шаблоны определяют семантику нагрузки: в каталоге предусмотрено шесть встроенных шаблонов~--- только чтение, только запись, смешанная лёгкая нагрузка, смешанная тяжёлая нагрузка, OLTP и аналитические запросы. Встроенные шаблоны создаются автоматически при инициализации системы. Пять шаблонов, кроме OLTP, поддерживают автоматическую генерацию наборов сценариев нагрузки на основе анализа схемы~--- система извлекает метаданные таблиц и строит параметризованные SQL-запросы для каждого такого шаблона. Шаблон OLTP требует ручного определения набора транзакций: каждая операция задаётся явно в виде последовательности связанных инструкций изменения данных, что обеспечивает точное моделирование транзакционной нагрузки без автогенерации по схеме.

Средний уровень образует профиль схемы~--- описание структуры данных, общей для группы совместимых баз данных. Для подключения в поле \texttt{profile\_source} фиксируется источник привязки: значение \texttt{auto} соответствует автоматическому определению, \texttt{inherited}~--- наследованию из группы, \texttt{manual}~--- ручному назначению, \texttt{pending\_review}~--- ожиданию подтверждения; для группы баз данных в поле \texttt{profile\_status} хранится статус профиля~--- черновик, подтверждён или несовместим. При подключении новой базы данных система автоматически анализирует её схему и предлагает наиболее подходящий профиль; коэффициент уверенности записывается в поле \texttt{profile\_confidence}.

Верхний уровень образует набор сценария нагрузки~--- конкретная связка профиля схемы и шаблона сценария. В зависимости от типа шаблона конфигурация содержит перечень параметризованных SQL-запросов для шаблонов чтения, записи, смешанных и аналитических сценариев либо набор транзакций с явно заданными последовательностями инструкций изменения данных для шаблона OLTP, а также спецификацию индексов. При запуске теста конфигурация фиксируется в неизменяемом снимке, который сохраняется вместе с прогоном, обеспечивая воспроизводимость результатов.

Группа баз данных объединяет набор физических подключений под единой моделью данных и связывает их с конкретным профилем схемы. Это позволяет тестировать несколько экземпляров одной и той же схемы~--- например, PostgreSQL, MySQL и MariaDB с одинаковой структурой данных~--- в рамках одного прогона без дополнительной настройки сценариев. Система отслеживает совместимость схем подключённых баз данных и формирует отчёт о расхождениях при их обнаружении.

Такой слой позволяет добавлять новые схемы через профиль и наборы сценариев без изменения ядра генерации нагрузки.

\subsection{Подсистема визуализации и сравнительного анализа}

Подсистема визуализации предназначена для представления результатов нагрузочного тестирования в наглядном виде и обеспечения возможности сравнительного анализа нескольких тестовых прогонов.

Модуль сравнительного анализа поддерживает два режима работы. Режим \texttt{per\_test} предназначен для попарного сравнения СУБД в рамках одного прогона с несколькими подключениями: выполняется ранжирование СУБД по метрикам задержки и пропускной способности. Режим \texttt{series} ориентирован на серийный анализ от двух до пяти прогонов с различными уровнями нагрузки: строятся траектории изменения метрик и оцениваются характеристики масштабируемости~--- деградация хвостовых процентилей, эластичность пропускной способности, точка насыщения; тренды проверяются критериями Спирмена и Манна--Кендалла. Режим задаёт пользователь на странице истории тестов и передаёт в запросе \texttt{POST /api/comparison/analyze} полем \texttt{analysis\_mode}.

Сравнение выборок задержки выполняется с предварительной проверкой нормальности по критерию Шапиро--Уилка: при нормальном распределении применяется критерий Уэлча, при ненормальном~--- критерий Манна--Уитни. Такой подход обоснован характером распределения задержек: эмпирические распределения времени отклика, как правило, содержат выбросы и не всегда близки к нормальным. Помимо статистической значимости, вычисляется размер эффекта~--- коэффициент Коэна~$d$,~--- позволяющий разграничить статистически значимые, но практически пренебрежимые различия от различий, имеющих прикладную ценность. При попарном сравнении нескольких СУБД применяется поправка на множественные сравнения для контроля уровня ложных открытий.

Аналитический отчёт формируется по структурированным правилам. В режиме \texttt{per\_test} он включает краткое резюме с высокоуровневым вердиктом, попарные сравнения, секцию практической значимости различий и конкретные рекомендации. В режиме \texttt{series} отчёт дополняется блоком \texttt{parameter\_impacts} с анализом влияния различий конфигурации между прогонами; в обоих режимах автоматически выявляются характерные паттерны нестабильности производительности, деградации при масштабировании и высоких хвостовых задержек.

Для визуализации результатов сравнения предусмотрены три типа диаграмм: столбчатая диаграмма абсолютных значений задержки по среднему времени отклика и перцентилям p95 и p99, диаграмма размаха для оценки разброса и асимметрии распределений, а также временной график пропускной способности для выявления нестабильности и трендов.

\subsection{Модель данных}

Хранение данных системы организовано в базе данных PostgreSQL. Модель включает 15~сущностей, разделённых на три слоя: слой тестовых результатов, слой конфигураций подключений и групп баз данных, а также слой сценариев и профилей схем.

Слой тестовых результатов содержит четыре сущности. TestRun~--- запись о тестовом прогоне~--- хранит уникальный идентификатор, наименование, статус, конфигурацию теста в формате JSON: типы СУБД, число виртуальных пользователей, число итераций, время прогрева, идентификатор сценария и итоговую статистику, а также ссылку на группу баз данных и поля управления состоянием: признак наличия write-операций, статус и результат верификации восстановления. Сущность связана с остальными тремя через внешние ключи с каскадным удалением. TestResult хранит агрегированные метрики производительности для одной СУБД в рамках прогона в формате JSON: полный набор показателей времени отклика (среднее, p50, p95, p99, минимум, максимум, стандартное отклонение), успешную и полную пропускную способность с учётом \texttt{workload\_mode}, при транзакционной нагрузке~--- \texttt{qps} и счётчики транзакций, частоту ошибок, итоговый снимок системных метрик хоста и внутренних метрик СУБД. TimeSeries содержит секундные точки временного ряда: время отклика, успешную пропускную способность, загрузку CPU, использование RAM (проценты и Мбайт), дисковые IOPS и скорости чтения/записи, сетевой трафик~--- для графиков реального времени. MetricSample хранит сырые индивидуальные замеры задержки, необходимые для статистического анализа: в отличие от TimeSeries, каждая запись фиксирует отдельное наблюдение~--- замер SQL-запроса или транзакции; фаза прогрева и признак ошибки задаются полями \texttt{measurement\_phase} и \texttt{is\_error}.

Слой конфигураций подключений включает две сущности. DatabaseConnectionConfig~--- конфигурация подключения к СУБД~--- содержит тип СУБД, адрес сервера, порт, имя базы данных, зашифрованный пароль, ссылку на профиль схемы, а также поля \texttt{detected\_profile\_name}, \texttt{profile\_confidence} и \texttt{profile\_source}. DatabaseGroup объединяет несколько физических подключений под единой моделью данных: хранит ссылку на профиль схемы, ссылку на эталонное подключение, поля \texttt{profile\_status} и \texttt{compatibility\_status}, а также детальный отчёт о расхождениях схем.

Слой сценариев и профилей включает девять сущностей. SchemaProfile~--- профиль схемы данных~--- описывает структуру, общую для группы совместимых баз данных, и связывает подключения с наборами сценариев нагрузки. ScenarioTemplate задаёт логический шаблон вида нагрузки без привязки к схеме. ScenarioBundle объединяет профиль схемы и шаблон; уникальность активной пары <<профиль + шаблон>> обеспечивается частичным уникальным индексом. ScenarioBundleQuery, ScenarioBundleParam и ScenarioBundleIndex хранят SQL-запросы, параметры генерации и спецификации индексов сценария соответственно. Для поддержки транзакционной нагрузки OLTP-класса предусмотрены ещё три сущности: ScenarioBundleTransaction~--- заголовочная запись транзакции с именем и весом, ScenarioBundleTransactionStep~--- отдельная SQL-инструкция внутри транзакции с порядковым номером и описанием, ScenarioBundleTransactionParam~--- параметр инструкции транзакции с типом генерации значения. Эти три сущности зеркально повторяют структуру сущностей для SQL-запросов применительно к транзакционной нагрузке и обеспечивают атомарное выполнение последовательностей операций изменения данных.

Все основные сущности используют UUID в качестве первичных ключей для обеспечения глобальной уникальности идентификаторов. Высоконагруженные таблицы TimeSeries и MetricSample используют целочисленный первичный ключ с автоинкрементом, что обеспечивает максимальную производительность вставки при потоковой записи метрик.
