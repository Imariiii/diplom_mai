\section{ПРОЕКТИРОВАНИЕ СИСТЕМЫ}

\subsection{Архитектура системы}

Разрабатываемая система построена по многоуровневой клиент-серверной архитектуре с разделением на серверную и клиентскую части. Серверная часть реализована на языке Python с использованием фреймворка FastAPI и обеспечивает генерацию нагрузки, сбор метрик, управление подключениями к тестируемым СУБД, хранение истории тестов, а также управление профилями схем и логическими базами данных. Клиентская часть реализована на TypeScript с использованием фреймворка Next.js и предоставляет веб-интерфейс для управления тестированием, визуализации метрик в реальном времени и анализа результатов.

Взаимодействие между клиентской и серверной частями осуществляется по двум каналам. REST API используется для выполнения CRUD-операций с тестами, сценариями, подключениями, логическими базами данных и профилями схем, а также для запуска тестов и получения результатов. WebSocket обеспечивает двустороннюю связь для трансляции метрик производительности в реальном времени в ходе выполнения теста. Такая архитектура позволяет клиенту инициировать тестирование через HTTP-запрос и затем наблюдать за его ходом через WebSocket-соединение без необходимости периодического опроса сервера.

В процессе работы система взаимодействует с базами данных двух типов. Собственная база данных системы предназначена для хранения истории тестов, сценариев, конфигураций подключений и результатов. Вторая группа — это целевые базы данных, над которыми непосредственно проводятся нагрузочные тесты. Такое структурное разделение позволяет изолировать служебные данные от тестируемых сред и исключить искажение результатов из-за операций сохранения метрик.

Архитектура системы включает следующие основные компоненты:

\begin{enumerate}[label=\arabic*)]
    \item \textbf{Модуль генерации нагрузки} — ядро системы, реализующее параллельное выполнение SQL-запросов с заданным числом виртуальных пользователей, сбор метрик производительности и управление жизненным циклом теста;
    \item \textbf{Модуль управления индексами} — создание и удаление индексов базы данных до и после тестового прогона в соответствии со спецификацией конфигурации сценария нагрузки;
    \item \textbf{Модуль управления подключениями} — абстракция подключения к различным СУБД, управление пулами соединений, формирование строк подключения и проверка доступности баз данных;
    \item \textbf{Модуль управления состоянием БД} — создание резервных копий перед тестами с операциями записи, восстановление состояния после завершения теста и верификация целостности данных;
    \item \textbf{Слой профилей и логических баз данных} — управление шаблонами сценариев, профилями схем, конфигурациями сценариев нагрузки и логическими базами данных, объединяющими подключения под единой моделью данных;
    \item \textbf{Модуль визуализации} — генерация графиков на основе временных рядов метрик, формирование сравнительных отчётов с применением статистических критериев;
    \item \textbf{Модуль хранения} — репозитории для работы с базой данных истории, реализующие операции создания, чтения, обновления и удаления записей о тестах, результатах, сценариях и профилях;
    \item \textbf{Модуль сравнительного анализа} — сервис для сопоставления результатов нескольких тестовых прогонов, вычисления описательной статистики и применения статистических критериев.
\end{enumerate}

Каждый компонент реализован в виде отдельного модуля с чётко определённым интерфейсом взаимодействия, что обеспечивает возможность независимой модификации и тестирования компонентов. Архитектура системы допускает расширение перечня поддерживаемых СУБД путём добавления новых драйверов подключения без модификации остальной кодовой базы.

Взаимодействие компонентов системы представлено на диаграмме (рисунок~\ref{fig:architecture}). Клиентский браузер отправляет HTTP-запрос на запуск теста, сервер создаёт запись в базе данных истории и запускает фоновую задачу. Фоновая задача вызывает \texttt{LoadTester}, который через \texttt{DatabaseConnection} подключается к тестируемым СУБД, выполняет SQL-запросы и собирает метрики. Метрики передаются через callback-механизм в \texttt{WebSocket\-Manager}, который рассылает их через WebSocket всем подключённым клиентам и сохраняет в базе данных истории.

\subsection{Проектирование модуля генерации нагрузки и сбора метрик}

Модуль генерации нагрузки является центральным компонентом системы и реализован в классе \texttt{LoadTester}. Его проектирование определялось необходимостью обеспечения параллельного выполнения SQL-запросов, точного замера времени отклика, сбора системных и внутренних метрик СУБД, управления индексами сценария, а также автоматического управления состоянием тестируемых баз данных.

Генерация нагрузки осуществляется с использованием асинхронной модели выполнения на базе библиотеки asyncio. Каждый виртуальный пользователь реализуется как асинхронная задача (\texttt{asyncio.Task}), выполняющая заданное число итераций SQL-запросов. Параллельное выполнение задач обеспечивается функцией \texttt{asyncio.gather()}, которая запускает все задачи одновременно и ожидает их завершения. Такой подход позволяет эффективно использовать ресурсы сервера без создания избыточного числа потоков операционной системы.

Для каждого выполняемого запроса фиксируется время отклика с использованием функции \texttt{time.perf\_counter()}, обеспечивающей высокую точность измерений. Результаты отдельных запросов накапливаются в структуре данных, из которой впоследствии вычисляются агрегированные метрики: среднее время отклика, медиана, процентили p50, p95, p99, минимальное и максимальное значения, а также количество ошибок.

Сбор метрик производительности организован в виде отдельного асинхронного цикла, выполняющегося параллельно с генерацией нагрузки. Цикл активируется с интервалом в одну секунду и на каждой итерации выполняет следующие действия:

\begin{enumerate}[label=\arabic*)]
    \item вычисляет агрегированные метрики за прошедший интервал времени на основе накопленных результатов запросов;
    \item собирает системные метрики через библиотеку psutil: загрузку центрального процессора, использование оперативной памяти, интенсивность дискового ввода-вывода, сетевой трафик;
    \item собирает внутренние метрики СУБД: коэффициент попаданий в кэш (cache hit ratio), количество ожиданий блокировок, количество взаимных блокировок (deadlocks), размеры таблиц;
    \item передаёт собранные метрики через callback-механизм для трансляции в реальном времени и сохранения в базе данных истории.
\end{enumerate}

Для трансляции метрик в реальном времени используется callback-паттерн. Класс \texttt{LoadTester} поддерживает два независимых callback: callback метрик (\texttt{set\_streaming\_callback}) и callback статуса операций резервного копирования (\texttt{set\_backup\_callback}). Конкретная реализация callback получает метрики от \texttt{LoadTester} и передаёт их в \texttt{WebSocketManager} для рассылки через WebSocket подключённым клиентам. Одновременно метрики сохраняются в базе данных истории как точки временного ряда (\texttt{TimeSeries}) и как сырые выборки (\texttt{MetricSample}).

\subsubsection*{Модуль управления индексами}

До начала основного этапа тестирования \texttt{IndexManager} создаёт индексы базы данных в соответствии со спецификацией выбранной конфигурации сценария нагрузки. Каждый индекс описывается объектом \texttt{ScenarioBundleIndex}, содержащим имя таблицы, перечень столбцов, тип индекса (btree по умолчанию), признак уникальности и необязательное условие (partial index). После завершения тестового прогона \texttt{IndexManager} удаляет созданные индексы, восстанавливая исходное состояние схемы. Результат каждой операции фиксируется в объекте \texttt{IndexOperationResult}, содержащем информацию о времени создания и удаления каждого индекса, а также список ошибок.

\subsubsection*{Алгоритм управления состоянием базы данных}

Перед выполнением теста с операциями записи (INSERT, UPDATE, DELETE) модуль генерации нагрузки инициирует создание резервной копии состояния базы данных. Алгоритм подготовки к тесту включает следующие шаги:

\begin{enumerate}[label=\arabic*)]
    \item \textbf{Анализ запросов:} модуль \texttt{QueryAnalyzer} просматривает все SQL-запросы сценария и определяет наличие операций записи. Если сценарий содержит только SELECT-запросы, резервное копирование не требуется;
    \item \textbf{Оценка размера:} определяется количество таблиц, которые будут затронуты операциями записи, и оценивается объём данных для резервного копирования. Если количество строк превышает порог в 1 миллион, пользователю выдаётся предупреждение;
    \item \textbf{Создание снимка состояния:} фиксируется количество строк в каждой таблице базы данных. Этот снимок будет использован для верификации после восстановления;
    \item \textbf{Создание резервной копии:} в зависимости от выбранной стратегии применяется один из двух механизмов резервного копирования.
\end{enumerate}

В системе реализованы две взаимозаменяемые стратегии резервного копирования. SQL-стратегия (используется по умолчанию) создаёт резервные копии таблиц средствами самой СУБД через запросы вида \texttt{CREATE TABLE AS SELECT}, размещая их с фиксированным префиксом \texttt{\_loadtest\_backup\_}. Данная стратегия работает через SQLAlchemy без привлечения внешних утилит и применима к любой поддерживаемой СУБД. Нативная стратегия вызывает утилиты командной строки: \texttt{mysqldump} для MySQL и \texttt{pg\_dump} для PostgreSQL, сохраняя результат в виде сжатого gzip-архива во временной директории. Нативная стратегия обеспечивает более полный дамп схемы, но требует наличия соответствующих клиентских утилит в окружении. Выбор стратегии задаётся параметром \texttt{default\_strategy} в конфигурации системы.

После завершения теста выполняется алгоритм восстановления:

\begin{enumerate}[label=\arabic*)]
    \item \textbf{Выполнение восстановления:} для SQL-стратегии данные копируются обратно из резервных таблиц; для нативной стратегии SQL-скрипт дампа выполняется через нативный клиент СУБД;
    \item \textbf{Верификация целостности:} количество строк в каждой таблице сравнивается с исходным снимком состояния. Если все значения совпадают, восстановление считается успешным;
    \item \textbf{Очистка:} резервные таблицы или временные файлы архива удаляются.
\end{enumerate}

Для предотвращения конфликтов при параллельном тестировании нескольких баз данных одного типа используется \texttt{asyncio.Lock} с отдельной блокировкой для каждой СУБД. Это гарантирует, что в один момент времени только один тест может выполнять операции backup/restore с конкретной СУБД.

\subsubsection*{Механизм параметризации запросов}

Для моделирования реалистичной нагрузки реализован механизм параметризации SQL-запросов. Каждый параметр запроса в конфигурации сценария нагрузки может быть сгенерирован одним из следующих способов:

\begin{enumerate}[label=\arabic*)]
    \item \texttt{random\_int} — случайное целое число из заданного диапазона [min, max];
    \item \texttt{random\_from\_table} — случайное значение из указанного столбца существующей таблицы базы данных;
    \item \texttt{sequential\_int} — последовательное целое число, инкрементируемое на каждой итерации;
    \item \texttt{uuid} — случайно сгенерированный UUID версии 4;
    \item \texttt{fixed} — фиксированное значение, заданное в конфигурации параметра;
    \item \texttt{random\_string} — случайная строка заданной длины из алфавитно-цифровых символов;
    \item \texttt{random\_date} — случайная дата из заданного диапазона.
\end{enumerate}

Такой подход позволяет варьировать данные в запросах и избегать эффектов кэширования планов выполнения, которые могут искажать результаты тестирования. Значения типа \texttt{random\_from\_table} кэшируются на уровне ключа \texttt{db\_key:table:column} для снижения числа дополнительных запросов к БД в ходе теста.

Перед основным этапом тестирования выполняется фаза прогрева (warmup), в ходе которой система выполняет серию предварительных запросов для инициализации кэшей и буферов СУБД. Это позволяет исключить влияние начальной инициализации на результаты основного этапа и получить более стабильные и воспроизводимые метрики производительности.

\subsection{Проектирование слоя сценариев и логических баз данных}

Для обеспечения гибкого сопоставления нагрузочных сценариев с конкретными базами данных в системе спроектирован трёхуровневый слой управления сценариями, включающий шаблоны сценариев, профили схем и конфигурации сценариев нагрузки. Данная архитектура позволяет описать логику нагрузки независимо от конкретной СУБД и автоматически подбирать конкретные SQL-запросы при выборе подключения.

Нижний уровень образует \textbf{шаблон сценария} (\texttt{ScenarioTemplate}~--- логическое описание вида нагрузки (например, смешанная нагрузка, только чтение, тяжёлые агрегации). Шаблон задаётся идентификатором, наименованием и описанием; встроенные шаблоны создаются при инициализации системы. Шаблоны не содержат конкретного SQL~--- они определяют семантику нагрузки.

Средний уровень образует \textbf{профиль схемы} (\texttt{SchemaProfile})~--- описание структуры данных, общей для группы совместимых баз данных (например, схема Sakila/Pagila). Профиль связывает набор подключений и конфигурацию сценария нагрузки с конкретной моделью данных. Поле \texttt{detection\_mode} задаёт режим автоопределения профиля (\texttt{hybrid}, \texttt{manual} и т.\,д.), что позволяет системе при подключении новой базы данных автоматически предложить соответствующий профиль на основе структуры её схемы. Степень уверенности определения фиксируется в полях \texttt{detected\_profile\_name} и \texttt{profile\_confidence} в записи \texttt{DatabaseConnectionConfig}.

Верхний уровень образует \textbf{конфигурация сценария нагрузки} (\texttt{ScenarioBundle})~--- конкретная связка профиля схемы и шаблона сценария. Конфигурация сценария содержит перечень SQL-запросов (\texttt{ScenarioBundleQuery}) с параметрами (\texttt{ScenarioBundleParam}) и спецификацию индексов (\texttt{ScenarioBundleIndex}). Уникальность пары <<профиль + шаблон>> обеспечивается частичным уникальным индексом по активным конфигурациям сценариев нагрузки. Это исключает дублирование и гарантирует, что для каждой пары существует не более одного активного варианта.

\textbf{Логическая база данных} (\texttt{LogicalDatabase}) объединяет набор физических подключений (\texttt{DatabaseConnectionConfig}) под единой логической моделью данных и связывает их с конкретным профилем схемы. Это позволяет тестировать несколько экземпляров одной и той же схемы в рамках одного прогона без дополнительной настройки сценариев. Привязка тестового прогона к логической базе данных фиксируется в поле \texttt{logical\_database\_id} записи \texttt{TestRun}.

При запуске системы класс \texttt{LogicalScenarioBootstrap} выполняет начальное заполнение: создаёт встроенные шаблоны сценариев и профили схем, а при наличии подключений с распознанными профилями~--- автоматически формирует соответствующие конфигурации сценариев нагрузки. Bootstrap выполняется однократно после применения миграций Alembic и не требует ручной инициализации.

Спроектированный трёхуровневый слой обеспечивает масштабируемость системы: добавление нового набора тестовых данных сводится к созданию профиля и генерации конфигураций сценариев нагрузки без изменения логики генерации нагрузки. Реализованная архитектура формирует основу для перехода к следующему компоненту системы~--- подсистеме визуализации и сравнительного анализа.

\subsection{Подсистема визуализации и сравнительного анализа}

Подсистема визуализации предназначена для представления результатов нагрузочного тестирования в наглядном виде и обеспечения возможности сравнительного анализа нескольких тестовых прогонов. Она включает два основных компонента: модуль генерации графиков и модуль сравнительного анализа.

Модуль сравнительного анализа реализован в классе \texttt{ComparisonService} и обеспечивает сопоставление результатов от двух до пяти тестовых прогонов. Класс принимает репозитории тестов, конфигурации сценариев нагрузки и подключений, что позволяет обогащать отчёт информацией о конфигурации сравниваемых прогонов. Процесс сравнения включает следующие этапы:

\begin{enumerate}[label=\arabic*)]
    \item загрузка данных о сравниваемых тестах из базы данных истории, включая конфигурации, результаты и сырые выборки метрик;
    \item автоматическое определение типа сравнения на основе сигнатур нагрузки: сравнение разных СУБД при одинаковой нагрузке (\texttt{CROSS\_DATABASE}), сравнение при разном числе потоков (\texttt{SCALABILITY}), сравнение при одинаковой конфигурации но разном времени выполнения (\texttt{TEMPORAL}), или смешанный сценарий (\texttt{MIXED});
    \item сбор пакета метрик для каждого теста: описательная статистика задержки (среднее, медиана, стандартное отклонение, минимум, максимум, p50, p95, p99), статистика пропускной способности, частота ошибок;
    \item нормализация метрик для обеспечения корректного сравнения тестов с разной конфигурацией: расчёт пропускной способности на один поток и эффективности масштабирования;
    \item попарное сравнение каждого теста с базовым (baseline) тестом с вычислением процентного отклонения метрик;
    \item применение статистических критериев для определения значимости различий;
    \item анализ влияния параметров конфигурации на метрики производительности;
    \item генерация структурированного аналитического отчёта с несколькими секциями.
\end{enumerate}

\subsubsection*{Алгоритм выбора статистического критерия}

Выбор статистического критерия осуществляется автоматически на основе проверки нормальности распределения. Алгоритм работает следующим образом:

\begin{enumerate}[label=\arabic*)]
    \item \textbf{Проверка размера выборки:} если количество элементов менее 10, статистический тест не выполняется~--- выводится предупреждение о недостаточном объёме данных;
    \item \textbf{Ограничение выборки:} если размер выборки превышает 5000 элементов, извлекается случайная подвыборка из 5000 элементов для снижения вычислительной нагрузки;
    \item \textbf{Проверка нормальности:} применяется критерий Шапиро-Уилка (\texttt{scipy.stats.shapiro}). Нулевая гипотеза~--- выборка имеет нормальное распределение. Если p-значение критерия больше 0,05, нулевая гипотеза не отвергается;
    \item \textbf{Выбор критерия:} при нормальном распределении применяется критерий Уэлча (\texttt{scipy.stats.ttest\_ind} с параметром \texttt{equal\_var=False}), который не требует равенства дисперсий. При ненормальном распределении применяется критерий Манна-Уитни (\texttt{scipy.stats.mannwhitneyu} с параметром \texttt{alternative="two-sided"}), являющийся непараметрическим аналогом t-критерия;
    \item \textbf{Интерпретация:} если p-значение выбранного критерия меньше 0,05, различие между выборками считается статистически значимым. Формируется текстовая интерпретация с указанием процентного отклонения и значения p.
\end{enumerate}

Для обеспечения корректности сравнения при отсутствии сырых выборок метрик (например, для тестов, выполненных до добавления модуля сравнения) реализован механизм fallback: если сырые выборки недоступны, используются агрегированные метрики из записей \texttt{TestResult} с соответствующим предупреждением о сниженной надёжности анализа.

\subsubsection*{Структура аналитического отчёта}

Класс \texttt{ComparisonReportGenerator} формирует многосекционный аналитический отчёт на основе правил (rule-based подход). Отчёт включает следующие секции, каждая из которых может быть включена или отключена в конфигурации \texttt{AnalysisReportConfig}:

\begin{enumerate}[label=\arabic*)]
    \item \textbf{Краткое резюме (executive summary):} высокоуровневые выводы о соотношении производительности сравниваемых тестов;
    \item \textbf{Основной вердикт:} итоговое заключение о лидере сравнения с указанием процентного преимущества;
    \item \textbf{Влияние параметров конфигурации:} при наличии различимых параметров (число потоков, длительность, сценарий) формируется анализ их влияния на ключевые метрики производительности;
    \item \textbf{Выявленные паттерны:} автоматическое определение характерных паттернов~--- нестабильности, деградации при масштабировании, высоких хвостовых задержек;
    \item \textbf{Практическая значимость различий (effect size):} интерпретация статистической значимости в практических терминах;
    \item \textbf{Рекомендации:} конкретные рекомендации по результатам анализа;
    \item \textbf{Гипотезы:} возможные причины наблюдаемых различий с пометкой о гипотетическом характере.
\end{enumerate}

\subsubsection*{Визуализация результатов сравнения}

Модуль сравнительного анализа формирует три типа визуализаций:

\begin{enumerate}[label=\arabic*)]
    \item \textbf{Столбчатая диаграмма (bar chart):} отображает среднее время отклика, p95 и p99 для каждого сравниваемого теста. Позволяет быстро оценить абсолютные различия в производительности;
    \item \textbf{Диаграмма размаха (box plot):} отображает five-number summary (минимум, первый квартиль, медиана, третий квартиль, максимум) распределения задержек для каждого теста. Позволяет оценить разброс и асимметрию распределения;
    \item \textbf{График пропускной способности во времени (throughput over time):} отображает изменение TPS на протяжении теста для каждого сравниваемого прогона. Позволяет выявить нестабильность производительности и тренды.
\end{enumerate}

\subsection{Модель данных}

Хранение данных системы организовано в базе данных PostgreSQL и включает следующие основные сущности.

\textbf{TestRun} — запись о тестовом прогоне. Содержит уникальный идентификатор (UUID), наименование, статус (pending, running, completed, failed), конфигурацию теста в формате JSON (типы СУБД, число виртуальных пользователей, число итераций, время прогрева, идентификатор сценария), итоговую статистику в формате JSON, а также ссылку на логическую базу данных (\texttt{logical\_database\_id}) и поля управления резервным копированием: наличие операций записи, перечень затронутых таблиц, статус восстановления, длительность восстановления, ошибки восстановления, результат верификации. Связана с записями TestResult, TimeSeries и MetricSample через внешние ключи с каскадным удалением.

\textbf{TestResult} — результаты тестирования для одной СУБД в рамках прогона. Содержит идентификатор тестового прогона, тип СУБД, идентификатор запроса, агрегированные метрики производительности в формате JSON (среднее время отклика, медиана, p95, p99, TPS, количество ошибок, частота ошибок), системные метрики и внутренние метрики СУБД.

\textbf{TimeSeries} — точки временного ряда для построения графиков. Каждая запись соответствует одному секундному интервалу тестирования и содержит: идентификатор тестового прогона, тип СУБД, временную метку, значения метрик (время отклика, TPS, загрузка CPU, использование RAM, дисковые IOPS, сетевой трафик, активные соединения, счётчик ошибок). Составной индекс по полям (test\_run\_id, db\_type, timestamp) оптимизирует запросы на выборку данных для графиков.

\textbf{MetricSample} — сырые выборки метрик для статистического анализа. Каждая запись содержит: идентификатор тестового прогона, тип СУБД, ключ подключения, идентификатор запроса, тип выборки (request\_latency, throughput\_window), временную метку, задержку в миллисекундах, пропускную способность и признак ошибки. Составной индекс по полям (test\_run\_id, db\_type, timestamp) обеспечивает эффективную выборку для статистических тестов.

\textbf{DatabaseConnectionConfig} — конфигурация подключения к СУБД. Содержит тип СУБД, адрес сервера, порт, наименование базы данных, имя пользователя, зашифрованный пароль (алгоритм AES-256), группу подключений, идентификатор логической базы данных (\texttt{logical\_database\_id}), идентификатор профиля схемы (\texttt{schema\_profile\_id}), а также результаты автоопределения профиля: имя определённого профиля (\texttt{detected\_profile\_name}), коэффициент уверенности (\texttt{profile\_confidence}) и источник привязки (\texttt{profile\_source}: auto или manual).

\textbf{SchemaProfile} — профиль схемы данных, описывающий структуру, общую для группы совместимых баз данных. Содержит наименование, описание, режим определения (\texttt{detection\_mode}), ссылку на эталонное подключение, признак встроенного профиля. Связан с подключениями (\texttt{DatabaseConnectionConfig}), логическими базами данных и конфигурациями сценариев нагрузки.

\textbf{LogicalDatabase} — логическая база данных, объединяющая несколько физических подключений под единой моделью данных. Содержит наименование, описание и ссылку на профиль схемы. Связана с набором \texttt{DatabaseConnectionConfig} и с тестовыми прогонами через поле \texttt{logical\_database\_id} в \texttt{TestRun}.

\textbf{ScenarioTemplate} — логический шаблон сценария нагрузки (встроенный или пользовательский). Содержит строковый идентификатор, наименование, описание и признак встроенности. Связан с конфигурациями сценариев нагрузки через отношение <<один ко многим>>.

\textbf{ScenarioBundle} — конкретная конфигурация сценария нагрузки, объединяющий профиль схемы и шаблон сценария. Содержит наименование, описание, источник генерации (\texttt{generation\_source}), признаки встроенности и активности. Уникальность активной пары <<профиль + шаблон>> обеспечивается частичным уникальным индексом. Связан с перечнями SQL-запросов и индексов.

\textbf{ScenarioBundleQuery} — SQL-запрос внутри конфигурации сценария нагрузки. Содержит шаблон SQL-запроса, тип запроса (SELECT, INSERT, UPDATE, DELETE), вес (определяет вероятность выбора при смешанной нагрузке) и порядковый номер выполнения. Связан с параметрами через отношение <<один ко многим>>.

\textbf{ScenarioBundleParam} — параметр SQL-запроса внутри конфигурации сценария нагрузки. Содержит наименование параметра, тип генерации значений (random\_int, random\_from\_table, sequential\_int, uuid, fixed, random\_string, random\_date), диапазоны значений, ссылки на таблицу и столбец базы данных, текущее значение и шаг инкремента.

\textbf{ScenarioBundleIndex} — спецификация индекса, связанного с конфигурацией сценария нагрузки. Содержит имя таблицы, перечень столбцов, тип индекса (btree по умолчанию), наименование индекса, признак уникальности, условие (для частичных индексов) и описание. Используется \texttt{IndexManager} для создания и удаления индексов при выполнении тестового прогона.

Все основные сущности используют UUID в качестве первичных ключей для обеспечения глобальной уникальности идентификаторов и совместимости в распределённых сценариях. Высоконагруженные таблицы (\texttt{TimeSeries}, \texttt{MetricSample}) используют \texttt{BigInteger} для первичного ключа с автоинкрементом, что обеспечивает максимальную производительность вставки при потоковой записи метрик.
