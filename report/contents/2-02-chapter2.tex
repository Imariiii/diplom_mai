\section{ПРОЕКТИРОВАНИЕ СИСТЕМЫ}

В данной главе рассматривается процесс проектирования разрабатываемой системы сравнительного нагрузочного тестирования СУБД. Описывается архитектура системы, проектируются ключевые модули генерации нагрузки и сбора метрик, подсистема визуализации и сравнительного анализа, а также модель данных.

\subsection{Архитектура системы}

Разрабатываемая система построена по многоуровневой клиент-серверной архитектуре с разделением на серверную и клиентскую части. Серверная часть реализована на языке Python с использованием фреймворка FastAPI и обеспечивает генерацию нагрузки, сбор метрик, управление подключениями к тестируемым СУБД и хранение истории тестов. Клиентская часть реализована на TypeScript с использованием фреймворка Next.js и предоставляет веб-интерфейс для управления тестированием, визуализации метрик в реальном времени и анализа результатов.

Взаимодействие между клиентской и серверной частями осуществляется по двум каналам. REST API используется для выполнения CRUD-операций с тестами, сценариями и подключениями, а также для запуска тестов и получения результатов. WebSocket обеспечивает двустороннюю связь для трансляции метрик производительности в реальном времени в ходе выполнения теста. Такая архитектура позволяет клиенту инициировать тестирование через HTTP-запрос и затем наблюдать за его ходом через WebSocket-соединение без необходимости периодического опроса сервера.

Система взаимодействует с тремя базами данных. Две из них являются объектами тестирования — MySQL (с тестовой схемой Sakila) и PostgreSQL (с тестовой схемой Pagila). Третья база данных, реализованная на PostgreSQL, служит для хранения истории тестов, сценариев, конфигураций подключений и результатов. Такое разделение позволяет изолировать данные системы от тестируемых СУБД и избежать влияния операций хранения истории на результаты тестирования.

Архитектура системы включает следующие основные компоненты:

\begin{enumerate}[label=\arabic*)]
    \item \textbf{Модуль генерации нагрузки} — ядро системы, реализующее параллельное выполнение SQL-запросов с заданным числом виртуальных пользователей, сбор метрик производительности и управление жизненным циклом теста;
    \item \textbf{Модуль управления подключениями} — абстракция подключения к различным СУБД, управление пулами соединений, формирование строк подключения и проверка доступности баз данных;
    \item \textbf{Модуль управления состоянием БД} — создание резервных копий перед тестами с операциями записи, восстановление состояния после завершения теста и верификация целостности данных;
    \item \textbf{Модуль визуализации} — генерация графиков на основе временных рядов метрик, формирование сравнительных отчётов с применением статистических критериев;
    \item \textbf{Модуль хранения} — репозитории для работы с базой данных истории, реализующие операции создания, чтения, обновления и удаления записей о тестах, результатах и сценариях;
    \item \textbf{Модуль сравнительного анализа} — сервис для сопоставления результатов нескольких тестовых прогонов, вычисления описательной статистики и применения статистических критериев.
\end{enumerate}

Каждый компонент реализован в виде отдельного модуля с чётко определённым интерфейсом взаимодействия, что обеспечивает возможность независимой модификации и тестирования компонентов. Архитектура системы допускает расширение перечня поддерживаемых СУБД путём добавления новых драйверов подключения без модификации остальной кодовой базы.

Взаимодействие компонентов системы представлено на диаграмме (рисунок~\ref{fig:architecture}). Клиентский браузер отправляет HTTP-запрос на запуск теста, сервер создаёт запись в базе данных истории и запускает фоновую задачу. Фоновая задача вызывает \texttt{LoadTester}, который через \texttt{DatabaseConnection} подключается к тестируемым СУБД, выполняет SQL-запросы и собирает метрики. Метрики передаются через \texttt{TestStreamingCallback} в \texttt{ConnectionManager}, который рассылает их через WebSocket всем подключённым клиентам и сохраняет в базе данных истории.

\subsection{Проектирование модуля генерации нагрузки и сбора метрик}

Модуль генерации нагрузки является центральным компонентом системы и реализован в классе \texttt{LoadTester}. Его проектирование определялось необходимостью обеспечения параллельного выполнения SQL-запросов, точного замера времени отклика, сбора системных и внутренних метрик СУБД, а также автоматического управления состоянием тестируемых баз данных.

Генерация нагрузки осуществляется с использованием асинхронной модели выполнения на базе библиотеки asyncio. Каждый виртуальный пользователь реализуется как асинхронная задача (asyncio.Task), выполняющая заданное число итераций SQL-запросов. Параллельное выполнение задач обеспечивается функцией \texttt{asyncio.gather()}, которая запускает все задачи одновременно и ожидает их завершения. Такой подход позволяет эффективно использовать ресурсы сервера без создания избыточного числа потоков операционной системы.

Для каждого выполняемого запроса фиксируется время отклика с использованием функции \texttt{time.perf\_counter()}, обеспечивающей высокую точность измерений. Результаты отдельных запросов накапливаются в структуре данных, из которой впоследствии вычисляются агрегированные метрики: среднее время отклика, медиана, процентили p50, p95, p99, минимальное и максимальное значения, а также количество ошибок.

Сбор метрик производительности организован в виде отдельного асинхронного цикла, выполняющегося параллельно с генерацией нагрузки. Цикл активируется с интервалом в одну секунду и на каждой итерации выполняет следующие действия:

\begin{enumerate}[label=\arabic*)]
    \item Вычисляет агрегированные метрики за прошедший интервал времени на основе накопленных результатов запросов;
    \item Собирает системные метрики через библиотеку psutil: загрузку центрального процессора, использование оперативной памяти, интенсивность дискового ввода-вывода (IOPS), сетевой трафик;
    \item Собирает внутренние метрики СУБД: коэффициент попаданий в кэш (cache hit ratio), коэффициент попаданий в буферный пул, количество ожиданий блокировок, количество взаимных блокировок (deadlocks), размеры таблиц;
    \item Передаёт собранные метрики через callback-механизм для трансляции в реальном времени и сохранения в базе данных истории.
\end{enumerate}

Для трансляции метрик в реальном времени используется callback-паттерн. Класс \texttt{LoadTester} определяет интерфейс \texttt{StreamingCallback} с методами \texttt{on\_metrics}, \texttt{on\_status\_change}, \texttt{on\_test\_start}, \texttt{on\_test\_complete} и \texttt{on\_test\_error}. Конкретная реализация callback, класс \texttt{TestStreamingCallback}, получает метрики от \texttt{LoadTester} и передаёт их в \texttt{ConnectionManager} для рассылки через WebSocket подключённым клиентам. Одновременно метрики сохраняются в базе данных истории как точки временного ряда (TimeSeries) и как сырые выборки (MetricSample).

\subsubsection*{Алгоритм управления состоянием базы данных}

Перед выполнением теста с операциями записи (INSERT, UPDATE, DELETE) модуль генерации нагрузки инициирует создание резервной копии состояния базы данных. Алгоритм подготовки к тесту включает следующие шаги:

\begin{enumerate}[label=\arabic*)]
    \item \textbf{Анализ запросов:} модуль \texttt{QueryAnalyzer} просматривает все SQL-запросы сценария и определяет наличие операций записи. Если сценарий содержит только SELECT-запросы, резервное копирование не требуется;
    \item \textbf{Оценка размера:} определяется количество таблиц, которые будут затронуты write-операциями, и оценивается объём данных для резервного копирования. Если количество строк превышает порог в 1 миллион, пользователю выдаётся предупреждение;
    \item \textbf{Создание снимка состояния:} фиксируется количество строк в каждой таблице базы данных. Этот снимок (fingerprint) будет использован для верификации после восстановления;
    \item \textbf{Создание резервной копии:} для MySQL вызывается утилита \texttt{mysqldump}, для PostgreSQL — \texttt{pg\_dump}. Результат сжимается алгоритмом gzip и сохраняется во временной директории с уникальным идентификатором.
\end{enumerate}

После завершения теста выполняется алгоритм восстановления:

\begin{enumerate}[label=\arabic*)]
    \item \textbf{Загрузка резервной копии:} дамп распаковывается из gzip-архива;
    \item \textbf{Выполнение восстановления:} SQL-скрипт дампа выполняется через нативный клиент СУБД. Для MySQL используется команда \texttt{mysql}, для PostgreSQL — \texttt{psql};
    \item \textbf{Верификация целостности:} количество строк в каждой таблице сравнивается с исходным снимком состояния. Если все значения совпадают, восстановление считается успешным;
    \item \textbf{Очистка:} временные файлы резервной копии удаляются.
\end{enumerate}

Для предотвращения конфликтов при параллельном тестировании нескольких баз данных одного типа используется \texttt{asyncio.Lock} с отдельной блокировкой для каждой СУБД (MySQL и PostgreSQL). Это гарантирует, что в один момент времени только один тест может выполнять операции backup/restore с конкретной СУБД.

\subsubsection*{Механизм параметризации запросов}

Для моделирования реалистичной нагрузки реализован механизм параметризации SQL-запросов. Каждый параметр запроса может быть сгенерирован одним из следующих способов:

\begin{enumerate}[label=\arabic*)]
    \item \texttt{random\_int} — случайное целое число из заданного диапазона [min, max];
    \item \texttt{random\_from\_table} — случайное значение из указанного столбца существующей таблицы базы данных;
    \item \texttt{sequential\_int} — последовательное целое число, инкрементируемое на каждой итерации;
    \item \texttt{uuid} — случайно сгенерированный UUID версии 4;
    \item \texttt{fixed} — фиксированное значение, заданное в конфигурации параметра;
    \item \texttt{random\_string} — случайная строка заданной длины из алфавитно-цифровых символов;
    \item \texttt{random\_date} — случайная дата из заданного диапазона.
\end{enumerate}

Такой подход позволяет варьировать данные в запросах и избегать эффектов кэширования планов выполнения, которые могут искажать результаты тестирования. Например, при тестировании сценария \texttt{mixed\_light} параметры запросов генерируют случайные идентификаторы фильмов и актёров из таблиц схемы Sakila, что имитирует разнообразие пользовательских запросов в реальной системе.

Перед основным этапом тестирования выполняется фаза прогрева (warmup), в ходе которой система выполняет серию предварительных запросов для инициализации кэшей и буферов СУБД. Это позволяет исключить влияние начальной инициализации на результаты основного этапа и получить более стабильные и воспроизводимые метрики производительности.

\subsection{Проектирование подсистемы визуализации и сравнительного анализа}

Подсистема визуализации предназначена для представления результатов нагрузочного тестирования в наглядном виде и обеспечения возможности сравнительного анализа нескольких тестовых прогонов. Она включает два основных компонента: модуль генерации графиков и модуль сравнительного анализа.

Модуль генерации графиков создаёт визуализации на основе временных рядов метрик, сохранённых в базе данных истории. Для каждого тестового прогона формируются графики зависимости времени отклика, пропускной способности, загрузки центрального процессора и использования оперативной памяти от времени. Графики строятся с использованием библиотеки matplotlib и сохраняются в формате PNG для последующего отображения в веб-интерфейсе.

Модуль сравнительного анализа реализован в классе \texttt{ComparisonService} и обеспечивает сопоставление результатов от двух до пяти тестовых прогонов. Процесс сравнения включает следующие этапы:

\begin{enumerate}[label=\arabic*)]
    \item Загрузка данных о сравниваемых тестах из базы данных истории, включая конфигурации, результаты и сырые выборки метрик;
    \item Автоматическое определение типа сравнения на основе сигнатур нагрузки: сравнение разных СУБД при одинаковой нагрузке (CROSS\_DATABASE), сравнение при разном числе потоков (SCALABILITY), сравнение при одинаковой конфигурации, но разном времени выполнения (TEMPORAL), или смешанный сценарий (MIXED);
    \item Сбор пакета метрик для каждого теста: описательная статистика задержки (количество, среднее, медиана, стандартное отклонение, минимум, максимум, p50, p95, p99), статистика пропускной способности, частота ошибок;
    \item Нормализация метрик для обеспечения корректного сравнения тестов с разной конфигурацией: расчёт пропускной способности на один поток, пропускной способности в секунду, эффективности масштабирования;
    \item Попарное сравнение каждого теста с базовым (baseline) тестом с вычислением процентного отклонения метрик;
    \item Применение статистических критериев для определения значимости различий: критерий Уэлча (Welch's t-test) для выборок с нормальным распределением и критерий Манна-Уитни для выборок с ненормальным распределением;
    \item Генерация текстового аналитического отчёта с интерпретацией результатов.
\end{enumerate}

\subsubsection*{Алгоритм выбора статистического критерия}

Выбор статистического критерия осуществляется автоматически на основе проверки нормальности распределения. Алгоритм работает следующим образом:

\begin{enumerate}[label=\arabic*)]
    \item \textbf{Проверка размера выборки:} если количество элементов менее 10, статистический тест не выполняется — выводится предупреждение о недостаточном объёме данных;
    \item \textbf{Ограничение выборки:} если размер выборки превышает 5000 элементов, извлекается случайная подвыборка из 5000 элементов для снижения вычислительной нагрузки;
    \item \textbf{Проверка нормальности:} применяется критерий Шапиро-Уилка (\texttt{scipy.stats.shapiro}). Нулевая гипотеза — выборка имеет нормальное распределение. Если p-значение критерия больше 0,05, нулевая гипотеза не отвергается;
    \item \textbf{Выбор критерия:} при нормальном распределении применяется критерий Уэлча (\texttt{scipy.stats.ttest\_ind} с параметром \texttt{equal\_var=False}), который не требует равенства дисперсий сравниваемых выборок. При ненормальном распределении применяется критерий Манна-Уитни (\texttt{scipy.stats.mannwhitneyu} с параметром \texttt{alternative="two-sided"}), являющийся непараметрическим аналогом t-критерия;
    \item \textbf{Интерпретация:} если p-значение выбранного критерия меньше 0,05, различие между выборками считается статистически значимым. Формируется текстовая интерпретация: «Сравниваемый тест быстрее/медленнее на X \% (p = Y)».
\end{enumerate}

Для обеспечения корректности сравнения при отсутствии сырых выборок метрик (например, для тестов, выполненных до добавления модуля сравнения) реализован механизм fallback: если сырые выборки недоступны, используются агрегированные метрики из записей TestResult с соответствующим предупреждением о сниженной надёжности анализа.

\subsubsection*{Визуализация результатов сравнения}

Модуль сравнительного анализа формирует три типа визуализаций:

\begin{enumerate}[label=\arabic*)]
    \item \textbf{Столбчатая диаграмма (bar chart):} отображает среднее время отклика, p95 и p99 для каждого сравниваемого теста. Позволяет быстро оценить абсолютные различия в производительности;
    \item \textbf{Диаграмма размаха (box plot):} отображает five-number summary (минимум, первый квартиль, медиана, третий квартиль, максимум) распределения задержек для каждого теста. Позволяет оценить разброс и асимметрию распределения;
    \item \textbf{График пропускной способности во времени (throughput over time):} отображает изменение TPS на протяжении теста для каждого сравниваемого прогона. Позволяет выявить нестабильность производительности и тренды.
\end{enumerate}

\subsection{Модель данных}

Хранение данных системы организовано в базе данных PostgreSQL и включает следующие основные сущности.

\textbf{TestRun} — запись о тестовом прогоне. Содержит уникальный идентификатор (UUID), наименование, статус (pending, running, completed, failed), конфигурацию теста в формате JSON (тип СУБД, число виртуальных пользователей, число итераций, время прогрева, идентификатор сценария), итоговую статистику в формате JSON, а также поля для управления резервным копированием: наличие операций записи, перечень затронутых таблиц, статус восстановления, длительность восстановления, ошибки восстановления, результат верификации. Связана с записями TestResult, TimeSeries и MetricSample через внешние ключи.

\textbf{TestResult} — результаты тестирования для одной СУБД. Содержит идентификатор тестового прогона, тип СУБД, наименование базы данных, агрегированные метрики производительности в формате JSON (среднее время отклика, медиана, p95, p99, TPS, количество ошибок), системные метрики и внутренние метрики СУБД.

\textbf{TimeSeries} — точки временного ряда для построения графиков. Каждая запись соответствует одному секундному интервалу тестирования и содержит: идентификатор тестового прогона, тип СУБД, временную метку, значения метрик в формате JSON (время отклика, TPS, загрузка CPU, использование RAM, дисковые IOPS, сетевой трафик, коэффициент попаданий в кэш, количество блокировок). Составной индекс по полям (test\_run\_id, db\_type, timestamp) оптимизирует запросы на выборку данных для графиков.

\textbf{MetricSample} — сырые выборки метрик для статистического анализа. Каждая запись содержит: идентификатор тестового прогона, тип СУБД, временную метку, время отклика конкретного запроса. Составной индекс по полям (test\_run\_id, db\_type, sample\_type) обеспечивает эффективную выборку для статистических тестов.

\textbf{TestScenario} — сценарий тестирования. Содержит наименование, тип (встроенный или пользовательский), описание. Связан с записями ScenarioQuery и ScenarioParam.

\textbf{ScenarioQuery} — SQL-запрос в составе сценария. Содержит идентификатор сценария, шаблон SQL-запроса, тип запроса (SELECT, INSERT, UPDATE, DELETE), вес (вероятность выполнения), порядок выполнения.

\textbf{ScenarioParam} — параметр SQL-запроса. Содержит идентификатор запроса, наименование параметра, тип генерации значений (random\_int, random\_from\_table, sequential\_int, uuid, fixed, random\_string, random\_date), диапазоны значений, ссылки на таблицы базы данных.

\textbf{DatabaseConnectionConfig} — конфигурация подключения к СУБД. Содержит тип СУБД, адрес сервера, порт, наименование базы данных, имя пользователя, зашифрованный пароль, группу подключений. Пароли шифруются с использованием алгоритма AES-256.

Все сущности используют UUID в качестве первичных ключей для обеспечения глобальной уникальности идентификаторов. Связи между сущностями реализованы через внешние ключи с каскадным удалением: при удалении записи TestRun автоматически удаляются связанные записи TestResult, TimeSeries и MetricSample.

\subsection{Выводы по главе 2}

В ходе проектирования системы решены следующие задачи:

\begin{enumerate}[label=\arabic*)]
    \item Спроектирована многоуровневая клиент-серверная архитектура с разделением на серверную часть (FastAPI) и клиентскую часть (Next.js), взаимодействующие через REST API и WebSocket;
    \item Спроектирован модуль генерации нагрузки на основе асинхронного выполнения SQL-запросов с параллельной обработкой виртуальных пользователей, сбором системных и внутренних метрик СУБД, механизмом параметризации запросов и фазой прогрева;
    \item Спроектирован алгоритм управления состоянием базы данных, включающий анализ запросов, оценку размера, создание снимка состояния, резервное копирование через SQL-дамп, восстановление и верификацию целостности;
    \item Спроектирована подсистема визуализации и сравнительного анализа, включающая генерацию графиков на основе временных рядов, автоматическое определение типа сравнения, нормализацию метрик и применение статистических критериев (Welch's t-test, Mann-Whitney U) с автоматическим выбором на основе проверки нормальности;
    \item Спроектирована модель данных, включающая сущности TestRun, TestResult, TimeSeries, MetricSample, TestScenario, ScenarioQuery, ScenarioParam и DatabaseConnectionConfig с оптимизированными индексами и каскадным удалением.
\end{enumerate}

Результаты проектирования определяют структуру и взаимодействие компонентов системы и служат основой для её программной реализации, рассматриваемой в следующей главе.
