\conclusion

В ходе выполнения выпускной квалификационной работы была разработана и реализована система для сравнительного нагрузочного тестирования и визуального анализа производительности реляционных баз данных. Все четыре задачи, поставленные во введении, решены в полном объёме, а цель работы --- создание программного комплекса, обеспечивающего автоматизированное тестирование, сбор метрик и их наглядное представление в едином веб-интерфейсе, --- достигнута.

В первой главе проведён аналитический обзор современных подходов к нагрузочному тестированию СУБД. Рассмотрены стандартизированные методологии TPC-C, TPC-H и TPC-E, определяющие эталонные сценарии транзакционной и аналитической нагрузки. Выполнен сравнительный анализ шести инструментов: HammerDB, Sysbench, pgbench, Oracle Real Application Testing, Benchmark Factory и Benchkit. Установлено, что ни одно из рассмотренных решений с открытым исходным кодом не обеспечивает одновременно генерацию нагрузки на несколько реляционных СУБД, мониторинг метрик через браузерный интерфейс в реальном времени, централизованное хранение истории тестов и автоматизированное сравнение прогонов с применением статистических критериев. На основании выявленного функционального разрыва сформулированы девять функциональных требований, нефункциональные требования и ограничения разрабатываемой системы.

Во второй главе спроектирована многоуровневая клиент-серверная архитектура, охватывающая серверную часть на базе FastAPI и клиентскую часть на базе Next.js. Разработан модуль генерации нагрузки, реализующий асинхронное выполнение SQL-запросов с параллельной обработкой виртуальных пользователей, сбором системных метрик и метрик СУБД, фазой прогрева и механизмом взвешенной параметризации запросов. Спроектированы два варианта стратегии резервного копирования --- SQL-based и Native --- с автоматическим восстановлением состояния базы данных после тестов с операциями записи и верификацией целостности через сравнение счётчиков строк. Введён слой логических баз данных и профилей схем: сущности \texttt{LogicalDatabase}, \texttt{SchemaProfile}, \texttt{ScenarioTemplate} и \texttt{ScenarioBundle} образуют механизм автоматической генерации пакетов запросов по метаданным реальных баз данных. Спроектирована подсистема сравнительного анализа с алгоритмом выбора статистического критерия (критерий Манна-Уитни или критерий Уэлча) в зависимости от распределения данных, вычислением размера эффекта Коэна~$d$, формированием структурированного аналитического отчёта и классификацией типа сравнения. Модель данных включает 13~сущностей, организованных в два независимых слоя: слой тестовых результатов и слой сценариев/профилей.

В третьей главе описана программная реализация системы. Серверная часть разработана на языке Python версии 3.9 и выше с применением фреймворка FastAPI, асинхронной ORM SQLAlchemy 2.0, драйверов asyncpg и aiomysql, библиотек Pydantic и pydantic-settings для валидации и конфигурации, а также инструмента Alembic для управления версиями схемы базы данных. Класс \texttt{LoadTester} реализует callback-механизм потоковой передачи метрик по WebSocket с частотой обновления одна секунда. Компонент \texttt{ScenarioBundleResolver} обеспечивает динамическое разрешение bundle-варианта для запускаемого теста с учётом профиля схемы и подключений. REST API объединяет девять групп эндпоинтов, охватывающих управление тестами, историей, подключениями, логическими базами данных, профилями схем и пакетами сценариев. Клиентская часть реализована на TypeScript с применением Next.js 16 (App Router), React 19, Zustand для управления состоянием, Recharts для визуализации, Tailwind CSS 4 и shadcn/ui; интерфейс включает девять страниц. Развёртывание системы организовано через Docker Compose, который запускает три сервиса: PostgreSQL-базу данных проекта, бэкенд и фронтенд. Инициализация схемы и заполнение встроенных данных полностью автоматизированы через Alembic и компонент \texttt{LogicalScenarioBootstrap}.

В четвёртой главе проведено тестирование системы на двух независимых наборах данных. На схеме Sakila/Pagila при одновременном нагрузочном тестировании MySQL, PostgreSQL и MariaDB установлено, что при низкой конкурентности (VU\,=\,20) все три СУБД демонстрируют близкие характеристики (средняя латентность 16,85--17,83~мс), тогда как при высокой конкурентности (VU\,=\,50) MariaDB обеспечивает наибольшую стабильность (p99\,=\,54~мс, $d = 2{,}47$), а PostgreSQL проявляет нестабильность хвоста распределения (p99 до~1~036~мс). На наборе Brazilian E-Commerce by Olist PostgreSQL устойчиво опережает MySQL и MariaDB на всех уровнях нагрузки: среднее время отклика ниже на~39--40\% при 10~VU. Для обоих наборов данных подтверждено, что число итераций в исследованных диапазонах не оказывает практически значимого влияния на характеристики производительности. Показатель ошибок запросов составил~0\,\% во всех 8 прогонах; система корректно классифицировала тип сравнения, применила статистические критерии и сформировала аналитические отчёты с интерпретацией и рекомендациями.

Разработанная система обеспечивает полный цикл сравнительного нагрузочного тестирования реляционных СУБД: от конфигурирования подключений и автоматической генерации пакетов запросов до мониторинга в реальном времени, хранения истории и формирования статистически обоснованных аналитических отчётов в едином веб-интерфейсе. Результаты работы применимы в задачах обоснованного выбора СУБД, оценки эффективности оптимизаций и предотвращения регрессии производительности. Перспективами дальнейшего развития системы являются поддержка дополнительных СУБД и NoSQL-хранилищ, реализация горизонтального масштабирования генераторов нагрузки, интеграция в CI/CD-пайплайны, а также расширение набора статистических методов анализа.
