\conclusion

В ходе выполнения выпускной квалификационной работы была разработана и реализована система для сравнительного нагрузочного тестирования и визуального анализа производительности реляционных баз данных. Все четыре задачи, поставленные во введении, решены в полном объёме, а цель работы --- создание программного комплекса, обеспечивающего автоматизированное тестирование, сбор метрик и их наглядное представление в едином веб-интерфейсе, --- достигнута.

В первой главе проведён аналитический обзор современных подходов к нагрузочному тестированию СУБД. Рассмотрены стандартизированные методологии TPC-C, TPC-H и TPC-E, определяющие эталонные сценарии транзакционной и аналитической нагрузки. Выполнен сравнительный анализ шести инструментов: HammerDB, Sysbench, pgbench, Oracle Real Application Testing, Benchmark Factory и Benchkit. Установлено, что ни одно из рассмотренных решений с открытым исходным кодом не обеспечивает одновременно генерацию нагрузки на несколько реляционных СУБД, мониторинг метрик через браузерный интерфейс в реальном времени, централизованное хранение истории тестов и автоматизированное сравнение прогонов с применением статистических критериев. На основании выявленного функционального разрыва сформулированы девять функциональных требований, нефункциональные требования и ограничения разрабатываемой системы.

Во второй главе спроектирована многоуровневая клиент-серверная архитектура, охватывающая серверную часть на базе FastAPI и клиентскую часть на базе Next.js. Разработан модуль генерации нагрузки, реализующий асинхронное выполнение SQL-запросов с параллельной обработкой виртуальных пользователей, сбором системных метрик и метрик СУБД, фазой прогрева и механизмом взвешенной параметризации запросов. Спроектированы два варианта стратегии резервного копирования --- SQL-based и Native --- с автоматическим восстановлением состояния базы данных после тестов с операциями записи и верификацией целостности через сравнение счётчиков строк. Введён слой логических баз данных и профилей схем: сущности \texttt{LogicalDatabase}, \texttt{SchemaProfile}, \texttt{ScenarioTemplate} и \texttt{ScenarioBundle} образуют механизм автоматической генерации пакетов запросов по метаданным реальных баз данных. Спроектирована подсистема сравнительного анализа с алгоритмом выбора статистического критерия в зависимости от распределения данных, вычислением размера эффекта Коэна~$d$, формированием структурированного аналитического отчёта и классификацией типа сравнения. Модель данных включает 13~сущностей, организованных в два независимых слоя: слой тестовых результатов и слой сценариев/профилей.

В третьей главе описана программная реализация системы. Серверная часть разработана на языке Python версии 3.9 и выше с применением фреймворка FastAPI, асинхронной ORM SQLAlchemy 2.0, драйверов asyncpg и aiomysql, библиотек Pydantic и pydantic-settings для валидации и конфигурации, а также инструмента Alembic для управления версиями схемы базы данных. Класс \texttt{LoadTester} реализует callback-механизм потоковой передачи метрик по WebSocket с частотой обновления одна секунда. Компонент \texttt{ScenarioBundleResolver} обеспечивает динамическое разрешение набора сценария нагрузки для запускаемого теста с учётом профиля схемы и подключений. REST API объединяет девять групп эндпоинтов, охватывающих управление тестами, историей, подключениями, логическими базами данных, профилями схем и пакетами сценариев. Клиентская часть реализована на TypeScript с применением Next.js 16, React 19, Zustand для управления состоянием, Recharts для визуализации, Tailwind CSS 4 и shadcn/ui; интерфейс включает девять страниц. Развёртывание системы организовано через Docker Compose, который запускает три сервиса: PostgreSQL-базу данных проекта, бэкенд и фронтенд. Инициализация схемы и заполнение встроенных данных полностью автоматизированы через Alembic и компонент \texttt{LogicalScenarioBootstrap}.

В четвёртой главе проведено тестирование системы на двух независимых наборах данных. На наборе Sakila при смешанной нагрузке MariaDB продемонстрировала наименьшую среднюю задержку и наибольшую пропускную способность, при этом её преимущество над PostgreSQL по латентности статистически значимо, однако эффект пренебрежимо мал. При анализе масштабируемости PostgreSQL показал наибольшую нестабильность хвоста распределения задержек --- деградация p95 составила $+437\%$, тогда как у MySQL и MariaDB рост был более умеренным. На наборе Brazilian E-Commerce при транзакционной нагрузке MySQL и MariaDB практически неразличимы, тогда как PostgreSQL уступает обеим СУБД по средней задержке на~32--34\%. Серийное тестирование на наборе Olist подтвердило достижение точки насыщения при~30~VU: пропускная способность MySQL и MariaDB практически не растёт, тогда как средняя задержка увеличивается в~4--5~раз. Показатель ошибок составил~0\,\% во всех~7~прогонах; система корректно применила статистические критерии и сформировала аналитические отчёты с ранжированием, попарными сравнениями и блоком влияния параметров нагрузки.

Разработанная система обеспечивает полный цикл сравнительного нагрузочного тестирования реляционных СУБД: от конфигурирования подключений и автоматической генерации пакетов запросов до мониторинга в реальном времени, хранения истории и формирования статистически обоснованных аналитических отчётов в едином веб-интерфейсе. Результаты работы применимы в задачах обоснованного выбора СУБД, оценки эффективности оптимизаций и предотвращения регрессии производительности. Перспективами дальнейшего развития системы являются поддержка дополнительных СУБД и NoSQL-хранилищ, реализация горизонтального масштабирования генераторов нагрузки, интеграция в CI/CD-пайплайны, а также расширение набора статистических методов анализа.
