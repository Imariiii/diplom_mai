\section{ТЕСТИРОВАНИЕ И АНАЛИЗ РЕЗУЛЬТАТОВ}

\subsection{Методика тестирования и тестовое окружение}

Тестирование разработанной системы проводилось на двух независимых наборах тестовых данных, охватывающих различные предметные области. Первый набор основан на схеме Sakila~\cite{sakila_db}~--- эталонной демонстрационной базе данных, разработанной командой MySQL. Второй~--- Brazilian E-Commerce Public Dataset~\cite{olist_dataset}, публичный набор данных электронной торговли с платформы Kaggle, содержащий сведения о реальных заказах.

Каждый набор данных был развёрнут в трёх экземплярах СУБД: MySQL~8, PostgreSQL~15 и MariaDB~10.x~--- в Docker-контейнерах. Серверная часть системы запускалась посредством Uvicorn, клиентская~--- через Next.js-сервер. База данных системы развёрнута в контейнере PostgreSQL~15 на порту~5433. Для схемы Sakila подключения работали на портах~3306 (MySQL), 5437 (PostgreSQL) и~3307 (MariaDB). Для набора Brazilian E-Commerce использовались порты~3308 (MySQL), 5434 (PostgreSQL) и~3309 (MariaDB).

Для совместного тестирования трёх экземпляров СУБД как единого объекта применялась функция группы баз данных~--- именованной группы физических подключений под единым профилем схемы. Система автоматически определила профиль \texttt{sakila\_like} для схемы Sakila и \texttt{olist\_like} для набора Brazilian E-Commerce, после чего сформировала наборы сценариев нагрузки, специфичные для каждого профиля. В экспериментах использованы три из шести встроенных шаблонов: смешанная лёгкая нагрузка, только чтение и OLTP; остальные шаблоны каталога (только запись, смешанная тяжёлая нагрузка, аналитические запросы) в прогонах не задействовались.

Тестирование выполнялось в двух форматах. В формате одиночного прогона все три подключения тестировались одновременно при фиксированных параметрах нагрузки; результаты использовались для попарного сравнения СУБД. В формате серийного тестирования выполнялось несколько прогонов с различными уровнями нагрузки для анализа масштабируемости. Параметры нагрузки задавались числом виртуальных пользователей и числом итераций на пользователя; время прогрева во всех прогонах составляло~5~секунд.

Статистическая значимость различий оценивалась по следующему алгоритму. Вначале проверялось распределение выборок критерием Шапиро--Уилка. При нормальном распределении применялся t-критерий Уэлча (для неравных дисперсий), при ненормальном~--- критерий Манна--Уитни. Практическая значимость определялась по размеру эффекта Коэна~$d$: $d < 0{,}2$~--- пренебрежимый, $0{,}2 \leq d < 0{,}5$~--- малый, $0{,}5 \leq d < 0{,}8$~--- средний, $d \geq 0{,}8$~--- большой. При попарном сравнении трёх СУБД применялась поправка Бенджамини--Хохберга для контроля уровня ложных открытий. Аналитический отчёт содержит уровень значимости (p-value) и текстовую интерпретацию, например: «PostgreSQL быстрее MySQL на 15\%», «размер эффекта средний».

\subsubsection{Масштабируемость при росте нагрузки}

Для набора Brazilian E-Commerce серийное тестирование проводилось со сценарием только чтения на трёх уровнях нагрузки: 10, 20 и 40~виртуальных пользователей при 80~итерациях каждый. Набор сценария нагрузки включал шесть SELECT-операций на тех же таблицах с JOIN-запросами через FK-связи и агрегацией.

Результаты серийного тестирования на наборе Brazilian E-Commerce демонстрируют принципиально иное поведение по сравнению с Sakila. Уже при минимальном уровне нагрузки средняя задержка исчисляется сотнями миллисекунд, а пропускная способность всех трёх СУБД выходит на плато около~6~запросов/с~--- и при переходе к более высоким уровням нагрузки практически не меняется. Это объясняется характером рабочей нагрузки~--- тяжёлые аналитические SELECT-запросы с JOIN-операциями по ключам \texttt{CHAR(32)} на объёме около~1{,}5~млн строк~--- а не деградацией или ошибками СУБД. Секундные значения хвостовых задержек отражают насыщение рабочей нагрузки, достигшей предела пропускной способности при данных характеристиках данных.

При базовой нагрузке PostgreSQL показывает наименьшую среднюю задержку, что объясняется более эффективным планировщиком запросов при работе со строковыми ключами. При переходе к 40~VU отрыв PostgreSQL сохраняется, однако ни одна из СУБД не демонстрирует значимого прироста пропускной способности. Ни в одном прогоне ошибочных выполнений запросов зафиксировано не было.

\subsection{Транзакционная нагрузка и верификация резервного копирования}

Для проверки механизма резервного копирования и восстановления данных проведены два прогона транзакционной нагрузки~--- по одному на каждом наборе данных. Оба прогона выполнялись при 15~виртуальных пользователях и 50~итерациях. В качестве сценария нагрузки использовался вручную сконфигурированный набор OLTP-транзакций: каждый виртуальный пользователь выполнял атомарные последовательности операций изменения данных, а не одиночные SQL-запросы. Поскольку сценарий содержит write-операции, система автоматически активировала механизм резервного копирования.

Для набора Sakila набор транзакций включал операции с таблицами \texttt{payment} и \texttt{rental}: выборку и обновление записей по первичному ключу, вставку новой записи и откат при нарушении ограничений. Для набора Brazilian E-Commerce~--- операции с таблицами \texttt{olist\_customers\_dataset} и \texttt{olist\_orders\_dataset}: выборку, обновление статуса заказа и вставку записи о событии.

Перед началом каждого прогона система зафиксировала снимок состояния тестируемых баз данных. По завершении теста выполнила восстановление и верификацию целостности путём сравнения числа строк в каждой таблице с зафиксированными значениями. По результатам обоих прогонов восстановление подтверждено: счётчики строк во всех таблицах совпали с исходными значениями, расхождений не обнаружено. Оба прогона завершены без ошибок. Таким образом, механизм резервного копирования обеспечил полную воспроизводимость write-нагрузки: каждый повторный прогон стартует с идентичного состояния данных.

\subsection{Сводные результаты}

По итогам двенадцати прогонов, охвативших оба набора данных, три СУБД и все режимы работы системы, получены следующие ключевые результаты.

\begin{enumerate}[label=\arabic*)]
    \item На Sakila при смешанной нагрузке: при умеренной конкуренции все три СУБД практически неразличимы по средней задержке и пропускной способности, различия сосредоточены в хвосте~--- MariaDB показывает лучший p99 (38~мс). При высокой нагрузке лидером по пропускной способности становится PostgreSQL ($\approx$610~запросов/с).
    \item На Brazilian E-Commerce при смешанной нагрузке: PostgreSQL устойчиво лидирует по пропускной способности на обоих уровнях нагрузки ($\approx$699 и $\approx$681~запросов/с), что обусловлено более эффективной обработкой строковых первичных ключей.
    \item При серийном анализе на Sakila: рост p95 на~65--72\% между уровнями нагрузки соответствует теоретическому поведению систем с очередями; на Brazilian E-Commerce фиксируется насыщение рабочей нагрузки ($\approx$6~запросов/с) без деградации СУБД.
    \item При транзакционной нагрузке: оба прогона подтвердили корректность резервного копирования и восстановления~--- счётчики строк совпали с исходными значениями, что доказывает воспроизводимость write-сценариев.
    \item Доля ошибок во всех~12~прогонах составила~0\%.
\end{enumerate}

\subsection{Выводы по главе}

Экспериментальная проверка на двух наборах данных и трёх СУБД в 12~прогонах подтвердила работоспособность всех основных функциональных возможностей системы. Режим попарного сравнения корректно выявил зависимость лидерства СУБД от уровня нагрузки: при умеренной конкуренции на Sakila три СУБД практически неразличимы по средней задержке, а при высокой нагрузке преимущество PostgreSQL по пропускной способности становится статистически значимым. На наборе с тяжёлым аналитическим профилем (Brazilian E-Commerce) PostgreSQL устойчиво лидирует по пропускной способности на всех уровнях нагрузки. Серийный анализ выявил ожидаемую деградацию хвостовых процентилей при росте числа виртуальных пользователей на Sakila и паттерн насыщения на Brazilian E-Commerce. Механизм резервного копирования и восстановления данных обеспечил полную воспроизводимость write-сценариев: все прогоны завершены с верифицированным восстановлением и нулевой долей ошибок. Полученные результаты демонстрируют практическую ценность разработанной системы как инструмента для объективного сравнения реляционных СУБД в контролируемых условиях.
