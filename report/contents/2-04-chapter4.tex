\section{ТЕСТИРОВАНИЕ И АНАЛИЗ РЕЗУЛЬТАТОВ}

\subsection{Методика тестирования и тестовое окружение}

Тестирование разработанной системы проводилось на двух независимых наборах тестовых данных, охватывающих различные предметные области. Первый набор основан на схеме Sakila~\cite{sakila_db} --- эталонной демонстрационной базе данных, разработанной командой MySQL. Второй --- Brazilian E-Commerce Public Dataset by Olist~\cite{olist_dataset}, публичный набор данных электронной торговли с платформы Kaggle, содержащий сведения о реальных заказах.

Каждый набор данных был развёрнут в трёх экземплярах СУБД: MySQL~8, PostgreSQL~15 и MariaDB~10.x --- в Docker-контейнерах. Серверная часть системы запускалась посредством Uvicorn, клиентская --- через Next.js-сервер. База данных системы развёрнута в контейнере PostgreSQL~15 на порту~5433. Для схемы Sakila подключения работали на портах~3306 (MySQL), 5437 (PostgreSQL) и~3307 (MariaDB). Для набора Olist использовались порты~3308 (MySQL), 5434 (PostgreSQL) и~3309 (MariaDB).

Для совместного тестирования трёх экземпляров СУБД как единого объекта применялась функция логической базы данных --- именованной группы физических подключений под единым профилем схемы. Система автоматически определила профиль \texttt{sakila\_like} для схемы Sakila и \texttt{olist\_like} для набора Olist, после чего сгенерировала пакеты запросов, специфичные для каждого профиля.

Тестирование выполнялось в двух форматах. В формате одиночного прогона все три подключения тестировались одновременно при фиксированных параметрах нагрузки; результаты использовались для попарного сравнения СУБД. В формате серийного тестирования выполнялось несколько прогонов с различными уровнями нагрузки для анализа масштабируемости. Параметры нагрузки задавались числом виртуальных пользователей~(VU) и числом итераций на пользователя; время прогрева во всех прогонах составляло~5~секунд.

Статистическая значимость различий оценивалась критерием Манна--Уитни с поправкой Бенджамини--Хохберга для множественных сравнений. Практическая значимость определялась по размеру эффекта Коэна~$d$: $d < 0{,}2$ --- пренебрежимый, $0{,}2 \leq d < 0{,}5$ --- малый, $0{,}5 \leq d < 0{,}8$ --- средний, $d \geq 0{,}8$ --- большой.

Данные различия в характеристиках двух наборов позволяют исследовать поведение СУБД в принципиально разных условиях нагрузки.

\subsection{Тестирование на наборе данных Sakila}

\subsubsection{Сравнение СУБД при смешанной нагрузке}

Первый прогон на наборе Sakila выполнялся в режиме \texttt{per\_test} с параметрами: 10~виртуальных пользователей, 50~итераций на пользователя, сценарий \texttt{mixed\_light}. Автоматически сгенерированный пакет запросов содержал пять операций на таблице \texttt{payment}: выборку строки по первичному ключу, обновление временной метки, выборку по внешнему ключу \texttt{customer\_id}, JOIN-запрос с таблицей \texttt{customer} и инкремент поля \texttt{amount}. Всего выполнено~1~500 транзакций.

Все три СУБД завершили тест без ошибок. MariaDB продемонстрировала наилучшие результаты одновременно по двум ключевым метрикам: наименьшую среднюю задержку и наибольшую пропускную способность. PostgreSQL и MySQL показали близкие значения средней задержки и пропускной способности. Примечательно, что при близких средних значениях хвостовые показатели существенно расходятся: p99 PostgreSQL достигает 280,0~мс~--- в~4,7 раза хуже, чем у MariaDB, что указывает на наличие редких выбросов.

Попарный статистический анализ выявил значимое различие между PostgreSQL и MariaDB: MariaDB быстрее на~31,9\%. Различие между MySQL и MariaDB статистически незначимо. При этом различие между PostgreSQL и MySQL значимо. Таким образом, статистически подтверждённое преимущество MariaDB по латентности является практически пренебрежимым; реальное преимущество данной СУБД в данном прогоне выражается прежде всего в более высокой пропускной способности. Ресурсные метрики демонстрируют равномерную загрузку: CPU~--- 1,4--5,1\%, коэффициент попаданий в кэш~--- 99,04--99,86\%, взаимных блокировок не зафиксировано.

\subsubsection{Масштабируемость при росте нагрузки}

Для анализа масштабируемости проведено серийное тестирование в режиме \texttt{series}: два прогона сценария \texttt{read\_only} на том же наборе Sakila с уровнями нагрузки 5~VU~/ 30~итераций и 30~VU~/ 30~итераций. Пакет запросов включал шесть SELECT-операций: выборку строки по PK, выборку нескольких столбцов по PK, выборку по FK, JOIN с одной таблицей, JOIN с двумя таблицами и агрегацию \texttt{COUNT(*)}.

MariaDB демонстрирует аналогичный рост среднего значения, однако деградация p95 более умеренная, хотя p99 также вырос значительно. Ни в одном прогоне ошибок выполнения запросов зафиксировано не было.

Таким образом, при данном уровне нагрузки все три СУБД демонстрируют масштабирование пропускной способности, однако PostgreSQL показывает наибольшую нестабильность хвоста распределения задержек, тогда как MySQL и MariaDB ведут себя схожим образом с умеренной деградацией p95.

\subsection{Тестирование на наборе Brazilian E-Commerce}

\subsubsection{Сравнение СУБД при смешанной нагрузке}

Первый прогон на наборе Brazilian E-Commerce выполнялся в режиме \texttt{per\_test} с параметрами: 10~VU, 50~итераций, сценарий \texttt{mixed\_light}. Пакет запросов содержал пять операций: инкремент поля по PK в таблице \texttt{olist\_customers\_dataset}, выборку строки по PK из той же таблицы, вставку строки в \texttt{olist\_geolocation\_dataset}, выборку строк по FK из \texttt{olist\_order\_items\_dataset} и JOIN-запрос \texttt{olist\_orders\_dataset} с \texttt{olist\_customers\_dataset}. Всего выполнено~1~500 транзакций.

PostgreSQL и MySQL показали близкие значения средней задержки и пропускной способности. При этом p99 PostgreSQL достигает 293,87~мс --- в~4,6 раза хуже, чем у MariaDB.

Статистический анализ выявил значимое различие в паре PostgreSQL~--- MariaDB: MariaDB быстрее на~40,1\%. Различие MySQL~--- MariaDB также значимо: MariaDB быстрее на~31,7\%. Различие PostgreSQL~--- MySQL статистически незначимо. Таким образом, MariaDB является единственной СУБД с практически значимым преимуществом на данном наборе данных при смешанной нагрузке. Коэффициент попаданий в кэш PostgreSQL составил~92,21\% --- ниже, чем у MySQL и MariaDB, что обусловлено превышением объёма набора Olist над размером буферного пула контейнера. Ни одной ошибки выполнения запросов зафиксировано не было.

Ни в одном прогоне ошибочных выполнений запросов зафиксировано не было.
