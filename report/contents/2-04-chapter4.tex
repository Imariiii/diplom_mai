\section{ТЕСТИРОВАНИЕ И АНАЛИЗ РЕЗУЛЬТАТОВ}

\subsection{Методика тестирования и тестовое окружение}

Тестирование разработанной системы проводилось на двух независимых наборах тестовых данных, охватывающих различные предметные области. Первый набор основан на схеме Sakila~\cite{sakila_db}~--- эталонной демонстрационной базе данных, разработанной командой MySQL. Второй~--- Brazilian E-Commerce Public Dataset~\cite{olist_dataset}, публичный набор данных электронной торговли с платформы Kaggle, содержащий сведения о реальных заказах.

Нагрузочные прогоны и эксплуатация разработанной системы выполнялись локально на одной рабочей станции без выделенного кластера и без распределения компонентов по отдельным серверам. Генерация нагрузки, сбор метрик, веб-интерфейс, служебная база данных проекта и все сравниваемые экземпляры СУБД размещались на общем вычислительном узле, что обеспечивало сопоставимость результатов при попарном сравнении движков в рамках одного прогона. Сравнение СУБД в таких условиях отражает относительное поведение при конкуренции за одни и те же ресурсы центрального процессора, оперативной памяти и подсистемы ввода-вывода, а не абсолютные показатели промышленного стенда.

Эксперименты выполнялись на персональном компьютере с процессором Intel Core i7-12700H двенадцатого поколения и архитектурой x86\_64. На уровне хостовой операционной системы Windows рабочая станция оснащена 40~Гбайтами оперативной памяти. Непосредственно прогонам подлежала гостевая среда Ubuntu~24.04 LTS в подсистеме Windows для Linux второго поколения: для неё доступны десять физических ядер, двадцать логических процессоров и около 19~Гбайт оперативной памяти, поскольку WSL2 выделяет виртуальной машине часть ресурсов хоста, а не весь объём установленной памяти. Данные и контейнеры размещались на виртуальном дисковом томе объёмом около 1~Тбайт, предоставленном той же подсистемой.

На описанном стенде каждый набор данных был развёрнут в трёх экземплярах СУБД: MySQL~8, PostgreSQL~15 и MariaDB~10.x~--- в Docker-контейнерах. Серверная часть системы запускалась посредством Uvicorn, клиентская~--- через Next.js-сервер. База данных системы развёрнута в контейнере PostgreSQL~15 на порту~5433. Для схемы Sakila подключения работали на портах~3306 для MySQL, 5437 для PostgreSQL и~3307 для MariaDB. Для набора Brazilian E-Commerce использовались порты~3308 для MySQL, 5434 для PostgreSQL и~3309 для MariaDB.

Для совместного тестирования трёх экземпляров СУБД как единого объекта применялась функция группы баз данных~--- именованной группы физических подключений под единым профилем схемы. Система автоматически определила профиль \texttt{sakila\_like} для схемы Sakila и \texttt{olist\_like} для набора Brazilian E-Commerce, после чего сформировала наборы сценариев нагрузки, специфичные для каждого профиля. В экспериментах использованы три из шести встроенных шаблонов: смешанная лёгкая нагрузка, только чтение и OLTP; остальные шаблоны каталога~--- только запись, смешанная тяжёлая нагрузка и аналитические запросы~--- в прогонах не задействовались.

Тестирование выполнялось в двух режимах системы: \texttt{per\_test} и \texttt{series}. В режиме \texttt{per\_test} все три подключения группы тестировались одновременно при фиксированных параметрах нагрузки; результаты использовались для попарного сравнения СУБД. В режиме \texttt{series} выполнялось по три прогона с различными уровнями нагрузки на каждый логический набор для анализа масштабируемости. Параметры нагрузки задавались числом виртуальных пользователей и числом итераций на пользователя; время прогрева во всех прогонах составляло~5~секунд. Во всех двенадцати прогонах параметр \texttt{use\_indexes} был отключён: нагрузка выполнялась на базовой схеме данных без дополнительных индексов из спецификации набора сценария. Всего выполнено двенадцать прогонов по матрице демонстрационных экспериментов. На всём протяжении прогонов система фиксировала полный набор метрик, описанный в главе~2: время отклика с перцентилями и стандартным отклонением, пропускную способность, системные показатели хоста и внутренние метрики СУБД. Статистическая обработка результатов и формирование аналитических отчётов выполнялись встроенным модулем сравнительного анализа по алгоритму, описанному при проектировании и реализации в главах~2 и~3: проверка нормальности, критерии Уэлча и Манна--Уитни, размер эффекта Коэна~$d$, поправка Бенджамини--Хохберга при множественных сравнениях. В таблицах и тексте ниже для сопоставления движков использованы преимущественно средняя задержка, хвостовые перцентили и успешная пропускная способность; для сценариев с одиночными SQL-запросами она приведена в запросах в секунду, для OLTP-прогонов~--- в транзакциях в секунду.

\subsection{Тестирование на наборе данных Sakila}

\subsubsection{Сравнение СУБД при смешанной нагрузке}

Прогоны на наборе Sakila в режиме \texttt{per\_test} выполнялись со сценарием смешанной лёгкой нагрузки при 20 и 50~виртуальных пользователях, по 100~итераций каждый. Автоматически сгенерированный набор сценария нагрузки для профиля \texttt{sakila\_like} включал операции на таблицах \texttt{rental} и \texttt{payment}: обновление временно\'{й} метки в \texttt{rental}, вставку строки в \texttt{payment}, выборки по первичному и внешнему ключам, а также JOIN-запрос \texttt{rental} с таблицей \texttt{customer}.

При умеренной нагрузке, 20~VU, все три СУБД демонстрируют близкие средние задержки и пропускную способность~--- около~27~мс и~600~запросов/с соответственно. Различия между СУБД выражены прежде всего в хвосте распределения задержек: наилучший p99 у MariaDB~--- 38~мс, тогда как у PostgreSQL и MySQL~--- 88 и 56~мс.

MySQL при этом показывает наибольший p99~--- около~178~мс, что указывает на рост редких медленных запросов при увеличении конкуренции. Попарный статистический анализ подтвердил значимость различий между PostgreSQL и MySQL по пропускной способности; различия между MySQL и MariaDB признаны незначимыми.

\subsubsection{Масштабируемость при росте нагрузки}

Для анализа масштабируемости на наборе Sakila в режиме \texttt{series} выполнены три прогона сценария только чтения с уровнями нагрузки 10, 20 и 40~виртуальных пользователей при 80~итерациях каждый. Набор сценария нагрузки включал шесть SELECT-операций: выборку строки по первичному ключу, выборку нескольких столбцов, выборку по внешнему ключу, JOIN с одной таблицей, JOIN с двумя таблицами и агрегацию \texttt{COUNT(*)}.

Серийный анализ Sakila в режиме \texttt{series} показывает закономерное нарастание хвостовой задержки: p95 у всех трёх СУБД растёт приблизительно на~65--72\% при каждом переходе с~10 на~20 и с~20 на~40~виртуальных пользователей~--- типичный эффект очередей при увеличении конкуренции. Пропускная способность при этом также нарастает с уровнем нагрузки; на максимальном уровне, 40~VU, PostgreSQL сохраняет лидерство~--- около~585~запросов/с, MariaDB~--- около~380, MySQL~--- около~370. Рост хвоста отражает ожидаемую деградацию под нагрузкой, а не сбой СУБД.

\subsection{Тестирование на наборе Brazilian E-Commerce}

Набор сценария нагрузки для профиля \texttt{olist\_like} включал операции на таблицах \texttt{olist\_customers\_dataset}, \texttt{olist\_geolocation\_dataset}, \texttt{olist\_order\_items\_dataset} и \texttt{olist\_orders\_dataset}: инкремент поля и выборку по первичному ключу, вставку строки, выборку по внешнему ключу и JOIN через FK-связи.

\subsubsection{Масштабируемость при росте нагрузки}

Для набора Brazilian E-Commerce в режиме \texttt{series} выполнены три прогона сценария только чтения на уровнях нагрузки 10, 20 и 40~виртуальных пользователей при 80~итерациях каждый. Набор сценария нагрузки включал шесть SELECT-операций на тех же таблицах с JOIN-запросами через FK-связи и агрегацией.

Серийный анализ в режиме \texttt{series} на наборе Brazilian E-Commerce демонстрирует иное поведение по сравнению с Sakila. Уже при 10~VU средняя задержка исчисляется сотнями миллисекунд, а пропускная способность всех трёх СУБД выходит на плато около~6~запросов/с и при росте числа виртуальных пользователей до~40 практически не увеличивается. Это объясняется тяжёлыми SELECT с JOIN по ключам \texttt{CHAR(32)} на объёме около~1{,}5~млн строк, а не отказом СУБД: насыщение достигается пределом самой рабочей нагрузки. На всех уровнях нагрузки PostgreSQL сохраняет наименьшую среднюю задержку.

\subsection{Транзакционная нагрузка и верификация резервного копирования}

Для проверки механизма резервного копирования и восстановления данных проведены два прогона транзакционной нагрузки~--- по одному на каждом наборе данных. Оба прогона выполнялись при 15~виртуальных пользователях и 50~итерациях в режиме \texttt{workload\_mode=transaction}. В качестве сценария нагрузки использовался вручную сконфигурированный набор OLTP-транзакций: каждый виртуальный пользователь выполнял атомарные последовательности операций изменения данных, а не одиночные SQL-запросы; основной показатель пропускной способности в таких прогонах~--- успешные транзакции в секунду, дополнительно фиксировалась интенсивность SQL-шагов (запросов в секунду). Поскольку сценарий содержит write-операции, система автоматически активировала механизм резервного копирования.

Для набора Sakila набор транзакций включал операции с таблицами \texttt{payment} и \texttt{rental}: выборку и обновление записей по первичному ключу, вставку новой записи и откат при нарушении ограничений. Для набора Brazilian E-Commerce~--- операции с таблицами \texttt{olist\_customers\_dataset} и \texttt{olist\_orders\_dataset}: выборку, обновление статуса заказа и вставку записи о событии.

Перед началом каждого прогона система зафиксировала снимок состояния тестируемых баз данных. По завершении теста выполнила восстановление и верификацию целостности путём сравнения числа строк в каждой таблице с зафиксированными значениями. По результатам обоих прогонов счётчики строк во всех таблицах совпали с исходными значениями, расхождений не обнаружено.

\subsection{Выводы по главе}

Экспериментальная проверка в двенадцати прогонах на локальном одноузловом стенде, на двух логических наборах данных и трёх СУБД подтвердила работоспособность разработанной системы: группы баз данных и профили схем, режимы \texttt{per\_test} и \texttt{series}, сравнительный анализ с отчётами и резервное копирование при OLTP-нагрузке. По результатам сравнения лидерство СУБД зависит от профиля схемы и уровня нагрузки: на компактной схеме Sakila при умеренной конкуренции различия проявляются прежде всего в хвосте задержек, при высокой~--- в пользу PostgreSQL по пропускной способности; на тяжёлом наборе Brazilian E-Commerce PostgreSQL устойчиво опережает конкурентов, а серийный анализ фиксирует насыщение рабочей нагрузки, а не отказ движков. Все прогоны завершены без ошибок выполнения запросов; write-сценарии воспроизводимы за счёт верифицированного восстановления состояния БД. Полученные данные демонстрируют практическую ценность системы как инструмента объективного сравнения реляционных СУБД в контролируемых условиях.
