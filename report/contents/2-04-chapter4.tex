\section{ТЕСТИРОВАНИЕ И АНАЛИЗ РЕЗУЛЬТАТОВ}

\subsection{Методика тестирования и тестовые стенды}

Тестирование разработанной системы проводилось на двух независимых наборах тестовых данных, охватывающих различные предметные области и классы нагрузки. Первый набор основан на схеме Sakila~\cite{sakila_db} --- эталонной демонстрационной базе данных, разработанной командой MySQL и описывающей систему управления видеопрокатом. Второй набор --- Brazilian E-Commerce Public Dataset by Olist~\cite{olist_dataset}, публичный набор данных электронной торговли с платформы Kaggle, содержащий сведения о реальных заказах бразильского маркетплейса. Оба набора данных развёртывались в три экземпляра: MySQL, PostgreSQL и MariaDB, что позволяло в каждом прогоне одновременно оценивать производительность трёх СУБД на идентичной нагрузке.

Программный стенд для тестирования был организован следующим образом. Серверная часть системы запускалась посредством Uvicorn на порту~8000, клиентская часть --- через Next.js dev-сервер на порту~3000. База данных истории тестов развёрнута в контейнере PostgreSQL 15 Alpine на порту~5433. Тестируемые СУБД развёртывались в Docker-контейнерах согласно конфигурациям, описанным в разделе~3.5. Для схемы Sakila контейнеры работали на портах~3306 (MySQL), 5432 (PostgreSQL / Pagila~\cite{pagila_db}) и~3307 (MariaDB). Для набора Olist PostgreSQL доступен на порту~5434; MySQL и MariaDB развёртываются по аналогичной схеме.

Конфигурации тестовых прогонов задавались двумя параметрами нагрузки: числом виртуальных пользователей (VU) и числом итераций на пользователя. Для набора Sakila исследовались конфигурации с VU\,=\,20 и VU\,=\,50 при числе итераций 121 и 201. Для набора Olist применялись конфигурации с VU\,=\,10 и VU\,=\,25 при числе итераций 100 и 161. Меньшие значения VU для набора Olist обусловлены существенно более высокой латентностью запросов к данному набору данных, что при более высоких значениях VU привело бы к нереалистичному накоплению очереди. В обоих наборах применялся сценарный шаблон \texttt{read\_only} (только чтение) с автоматически сгенерированными конфигурациями сценариев нагрузки на основе метаданных эталонной базы данных соответствующей СУБД. Время прогрева составляло 5~секунд для каждого прогона.

Сравнительный анализ прогонов выполнялся встроенным модулем системы. Для каждой пары тестов (сравниваемый и базовый) вычислялись попарные различия метрик латентности и пропускной способности, применялся критерий Манна-Уитни для оценки статистической значимости (уровень значимости $\alpha = 0{,}05$) и рассчитывался размер эффекта Коэна~$d$. В обоих экспериментах сравнение имело тип \texttt{mixed}: тесты различались как по числу VU, так и по числу итераций, что системой идентифицируется как многопараметрический анализ чувствительности. Базовым тестом в каждом из экспериментов служил прогон с минимальными значениями нагрузки.

\subsection{Характеристика тестовых наборов данных}

\subsubsection{Схема Sakila / Pagila}

Схема Sakila разработана командой MySQL в 2006~году как демонстрационная база данных и с тех пор используется в качестве эталона для тестирования и обучения~\cite{sakila_db}. Схема содержит 16~таблиц, описывающих сущности видеопроката: фильмы, актёров, категории, языки, инвентарь, магазины, персонал, клиентов, адреса, аренды и платежи. Таблицы связаны внешними ключами с каскадными ограничениями, первичные ключи имеют целочисленный тип, что обеспечивает высокую скорость поиска по индексам. Схема Pagila~\cite{pagila_db} представляет собой точный порт Sakila для PostgreSQL с сохранением логической структуры и набора данных, адаптированный к синтаксису и типам данных PostgreSQL. В данном исследовании на все три СУБД (MySQL, PostgreSQL и MariaDB) был импортирован идентичный набор данных Sakila, что обеспечивало равнозначность условий кросс-СУБД сравнения.

Автоматически сгенерированный пакет запросов для сценария \texttt{read\_only} на профиле \texttt{sakila\_like} включает запросы на чтение по первичным ключам ключевых таблиц схемы: выборка одной записи из \texttt{payment} по \texttt{payment\_id} (вес~30), выборка нескольких атрибутов платежа по \texttt{payment\_id} (вес~20), выборка платежей клиента по \texttt{customer\_id} (вес~15) и запросы к таблицам \texttt{rental} и \texttt{film}. Параметры запросов генерируются случайным образом из реальных значений соответствующих столбцов, что имитирует реалистичную рабочую нагрузку.

\subsubsection{Brazilian E-Commerce Public Dataset by Olist}

Brazilian E-Commerce Public Dataset by Olist~\cite{olist_dataset} --- публично доступный набор данных, размещённый на платформе Kaggle компанией Olist, крупнейшей бразильской торговой площадкой. Набор содержит около 100~000~заказов, оформленных в период с 2016 по 2018~год, и охватывает полный жизненный цикл заказа: от оформления до доставки и отзыва покупателя. В составе набора --- 9~таблиц: \texttt{olist\_orders\_dataset}, \texttt{olist\_order\_items\_dataset}, \texttt{olist\_products\_dataset}, \texttt{olist\_sellers\_dataset}, \texttt{olist\_customers\_dataset}, \texttt{olist\_order\_payments\_dataset}, \texttt{olist\_order\_reviews\_dataset}, \texttt{olist\_geolocation\_dataset} и справочник категорий \texttt{product\_category\_name\_translation}. В отличие от Sakila, первичные ключи всех основных таблиц представлены строками в формате MD5-хэша (тип~\texttt{CHAR(32)}), что приводит к менее эффективному поиску по индексам по сравнению с целочисленными ключами. Внешние ключи установлены между основными таблицами заказов, товаров, продавцов и клиентов; связь с таблицей геолокации не регламентирована жёсткими ограничениями ввиду избыточности данных в исходном источнике.

Набор данных импортирован в три СУБД: PostgreSQL, MySQL и MariaDB. Пакет запросов для сценария \texttt{read\_only} на профиле \texttt{olist\_like} включает выборку клиента по первичному ключу из \texttt{olist\_customers\_dataset} (вес~30), выборку ряда атрибутов клиента (вес~20), выборку позиций заказа из \texttt{olist\_order\_items\_dataset} по \texttt{order\_id} (вес~15) и запросы к таблицам заказов и платежей. Строковые первичные ключи обусловливают иной паттерн сравнения при поиске, нежели у Sakila, что существенно влияет на характеристики производительности.

\subsection{Демонстрация работы системы}

Работа системы демонстрируется на примере полного цикла нагрузочного тестирования. На начальной странице пользователь получает краткое описание системы и переходит к конфигурации теста. На странице подключения к СУБД пользователь выбирает или создаёт логическую базу данных, назначает ей профиль схемы и указывает подключения к конкретным экземплярам СУБД.

На странице конфигурации теста пользователь задаёт параметры нагрузки: число виртуальных пользователей, количество итераций, время прогрева. Система автоматически разрешает пакет сценария по назначенному профилю и выбранному шаблону. При нажатии кнопки <<Запустить тест>> система выполняет POST-запрос к эндпоинту \texttt{test/async}, сохраняет запись о прогоне в базе истории и возвращает идентификатор теста. Пользователь перенаправляется на страницу дашборда.

На странице дашборда устанавливается WebSocket-соединение по адресу \texttt{/ws/test/\{test\_id\}}, через которое в реальном времени поступают метрики латентности, пропускной способности, загрузки CPU и использования RAM. Прогресс-бар отображает степень завершённости прогона. По завершении теста пользователь переходит на страницу истории тестов, где выбирает от двух до пяти прогонов для сравнительного анализа. На странице сравнения система генерирует аналитический отчёт: сводную таблицу метрик, попарные статистические сравнения с p-значениями и размерами эффекта, выявленные паттерны и итоговые рекомендации.

Примечательно, что пропускная способность всех трёх СУБД при росте VU снизилась незначительно: MySQL --- с~980 до~876~зап/с ($-$10,6\%), PostgreSQL --- с~917 до~752~зап/с ($-$18,0\%), MariaDB --- с~947 до~866~зап/с ($-$8,6\%). Коэффициент эффективности масштабирования (throughput per thread) при этом снизился до~0,33--0,37 от исходного уровня, что указывает на возникновение конкуренции за ресурсы при 50~VU.

Модуль сравнительного анализа системы зафиксировал характерный паттерн: при нагрузке ниже~50 потоков PostgreSQL демонстрирует незначительное преимущество по средней латентности, однако при нагрузке в~50~VU лидерство переходит к MariaDB, а MySQL занимает промежуточную позицию. Аналитический отчёт системы интерпретирует данное явление как точку пересечения производительности (\textit{performance crossover}) и рекомендует принимать решение о выборе СУБД с учётом ожидаемого уровня конкурентности.

\subsubsection{Влияние числа итераций}

При сравнении прогонов с 121 и 201 итерациями при VU\,=\,20 все три СУБД показали статистически незначимые или пренебрежимо малые различия по метрикам производительности. Средняя латентность MySQL изменилась на~0,1\% (p\,=\,0{,}0002 при $d = 0{,}001$, т.\,е. несмотря на формальную статистическую значимость, практический эффект пренебрежимо мал). Для MariaDB разница составила +7,4\% (p\,=\,0{,}97, статистически незначима). PostgreSQL демонстрировал незначительное улучшение $-$6,5\% (p\,<\,0{,}001, $d = 0{,}03$). Пропускная способность также оставалась стабильной. Таким образом, в диапазоне 121--201 итерации количество итераций не оказывает существенного влияния на характеристики производительности схемы Sakila.

При конфигурации VU\,=\,50, итераций\,=\,201 система выявила аномальное снижение пропускной способности PostgreSQL до~205,7~зап/с (против~752 при аналогичном VU и 121~итерации) с ростом p99 до~1~378~мс, что свидетельствует о деградации под совокупным влиянием высокой конкурентности и увеличенного объёма работы. Для MariaDB в данной конфигурации объём выборки латентности составил только 9~измерений, что порождает предупреждение о недостаточности данных; результаты для этой конфигурации следует трактовать с осторожностью.

Данная характеристика обусловлена совокупностью факторов: использованием строковых первичных ключей типа \texttt{CHAR(32)} вместо целочисленных, значительно большим объёмом данных (порядка~100~000~заказов против~16~тысяч записей в ключевых таблицах Sakila), а также отсутствием пользовательских индексов в базовой конфигурации. PostgreSQL продемонстрировал устойчивое преимущество над MySQL и MariaDB: средняя латентность PostgreSQL ниже в~1,66 и~1,63~раза соответственно при существенно более высокой пропускной способности~--- 11,89~зап/с против~7,08~зап/с (MySQL) и~7,03~зап/с (MariaDB). Характерно, что MySQL и MariaDB показали практически идентичные результаты в этой конфигурации.

Значения p95 и p99 всех СУБД при данной нагрузке существенно превышают среднее, что указывает на высокую вариативность времён отклика и длинный хвост распределения. Медиана PostgreSQL (406~мс) значительно ниже среднего (766~мс), что свидетельствует о наличии выбросов и нестабильности отдельных запросов.

\subsection{Сравнительная характеристика двух наборов данных}

Сопоставление результатов на двух наборах данных позволяет сформулировать ряд обобщений. Средняя латентность на наборе Olist превышает значения для схемы Sakila примерно в~45--75~раз (например, 766~мс против~16,85~мс для PostgreSQL при минимальной нагрузке), что объясняется строковыми первичными ключами, большим объёмом данных и отсутствием пользовательских индексов. Пропускная способность на наборе Olist на два порядка ниже: 7--12~зап/с против~870--980~зап/с.

Поведение при масштабировании нагрузки также различается. На схеме Sakila различные СУБД ведут себя по-разному: MariaDB сохраняет наименьший p99 при высокой нагрузке, тогда как PostgreSQL демонстрирует нестабильность хвоста распределения. На наборе Olist все три СУБД масштабируются схожим образом (рост латентности на~135--160\% при 2,5-кратном увеличении VU), тогда как пропускная способность остаётся практически неизменной, что указывает на конкурентное ограничение системы, а не на линейный рост нагрузки.

Система автоматически классифицировала оба сравнения как тип \texttt{mixed} и корректно идентифицировала варьируемые параметры: число VU, число итераций, а также активацию индексов. Ни в одном из прогонов зафиксировано не было ошибочных выполнений запросов: показатель ошибок составил~0\,\% для всех конфигураций, что подтверждает корректность работы модуля генерации нагрузки и механизма восстановления состояния базы данных.
