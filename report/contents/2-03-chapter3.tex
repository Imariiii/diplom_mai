\section{ПРОГРАММНАЯ РЕАЛИЗАЦИЯ}

\subsection{Выбор и обоснование технологического стека}

Выбор технологического стека определялся требованиями к производительности, поддерживаемости и удобству разработки. Для серверной части был выбран язык Python версии 3.9 и выше в сочетании с фреймворком FastAPI. Python обеспечивает высокую скорость разработки благодаря развитой экосистеме библиотек для работы с базами данных, статистического анализа и визуализации. FastAPI предоставляет современный асинхронный веб-фреймворк с автоматической генерацией документации API через OpenAPI, встроенной валидацией данных через Pydantic и высокой производительностью, сопоставимой с фреймворками на Node.js.

Для асинхронного выполнения запросов к СУБД выбраны драйверы asyncpg для PostgreSQL и aiomysql для MySQL. Оба драйвера обеспечивают неблокирующий ввод-вывод, что позволяет эффективно обрабатывать множество параллельных соединений в рамках одного потока операционной системы. В качестве ORM-слоя используется SQLAlchemy версии 2.0 с поддержкой асинхронных сессий, что обеспечивает типобезопасную работу с базой данных истории. Для управления миграциями схемы базы данных применяется инструмент Alembic, позволяющий описывать изменения схемы в виде версионируемых Python-скриптов и применять их автоматически при старте приложения. Конфигурация системы управляется через библиотеку pydantic-settings, загружающую параметры из переменных окружения и файла \texttt{.env}.

Для клиентской части выбран фреймворк Next.js версии 16 на базе React 19 с использованием TypeScript. Next.js обеспечивает клиентскую маршрутизацию через App Router и встроенную поддержку серверных компонентов. TypeScript гарантирует типобезопасность на всех уровнях клиентского приложения, снижая количество ошибок времени выполнения. Для управления состоянием приложения выбрана библиотека Zustand, отличающаяся минимальным объёмом кода и отсутствием необходимости в оборачивании приложения в провайдеры. Для визуализации данных используется библиотека Recharts, предоставляющая декларативный API для построения графиков на базе SVG. Стилизация реализована через Tailwind CSS версии 4 с использованием компонентов из библиотеки shadcn/ui, построенной поверх Radix UI.

Для развёртывания всех компонентов системы~--- базы данных, серверной и клиентской частей~--- используется Docker Compose с образом PostgreSQL 15 Alpine для базы данных и собственными Dockerfile для бэкенда и фронтенда, что обеспечивает воспроизводимость окружения и единообразный запуск на любой платформе.

\subsection{Реализация серверной части}

Серверная часть системы реализована в виде модульного приложения FastAPI. Точка входа находится в файле \texttt{main.py}, где инициализируется экземпляр приложения, настраивается CORS-мидлвэр для разрешения кросс-доменных запросов от фронтенда, регистрируются маршруты и создаётся WebSocket-эндпоинт.

Инициализация компонентов системы выполняется через модуль \texttt{initialize.py}, который создаёт экземпляры пяти репозиториев: \texttt{TestRepository}, \texttt{ConnectionRepository}, \texttt{ProfileRepository}, \texttt{ScenarioBundleRepository} и \texttt{LogicalDatabaseRepository}. Каждый репозиторий принимает URL базы данных истории и создаёт собственный пул асинхронных соединений через SQLAlchemy. Такой подход реализует паттерн внедрения зависимостей и обеспечивает централизованное управление жизненным циклом компонентов. При недоступности базы данных инициализация не прерывает запуск приложения~--- репозитории устанавливаются в \texttt{None}, а соответствующая функциональность корректно деградирует.

Запуск и остановка приложения управляются через менеджер контекста lifespan. При запуске в первую очередь автоматически применяются миграции Alembic командой \texttt{upgrade head}, что гарантирует актуальность схемы базы данных вне зависимости от версии развёртывания. После успешных миграций запускается \texttt{LogicalScenarioBootstrap}, который создаёт встроенные шаблоны сценариев и профили схем, а при наличии зарегистрированных подключений с определёнными профилями~--- автоматически генерирует соответствующие bundle-варианты. Оба этапа выполняются атомарно при каждом старте и идемпотентны: повторный запуск не создаёт дублирующих записей.

\subsubsection*{Модуль генерации нагрузки}

Модуль генерации нагрузки реализован в классе \texttt{LoadTester} файла \texttt{load\_tester/tester.py} объёмом более 1400 строк. Класс содержит методы для выполнения отдельных запросов, запуска одиночного теста и сравнительного теста для нескольких СУБД. Каждый метод следует единой структуре: подготовка базы данных, выполнение нагрузки, сбор метрик, восстановление состояния, верификация. Блок \texttt{try/finally} гарантирует выполнение восстановления даже при возникновении ошибок в ходе тестирования.

Базовый метод \texttt{execute\_query} выполняет отдельный SQL-запрос к указанной базе данных, замеряя время отклика через \texttt{time.perf\_counter()}. Ниже приведён фрагмент реализации:

\begin{figure}
    \begin{minipage}{\linewidth}
\begin{lstlisting}[language=Python]
async def execute_query(self, db_key, query, query_id):
    """Выполнение одного запроса с измерением времени"""
    start_time = time.perf_counter()
    error = None
    rows_count = 0
    db_type = self.db_connection.get_dbms_type(db_key)
    
    try:
        engine = await self.db_connection.get_engine_async(db_key)
        async with engine.connect() as conn:
            result = await conn.execute(text(query))
            rows_count = len(result.fetchall()) if result.returns_rows else 0
            await conn.commit()
    except Exception as e:
        error = str(e)
    
    end_time = time.perf_counter()
    execution_time = (end_time - start_time) * 1000
    
    return {
        'query_id': query_id,
        'execution_time_ms': execution_time,
        'rows_count': rows_count,
        'error': error,
        'timestamp': datetime.now(timezone.utc).isoformat()
    }
\end{lstlisting}
    \end{minipage}
    \caption{Реализация метода executeQuery класса LoadTester}
    \label{lst:execute_query}
\end{figure}

Использование \texttt{time.perf\_counter()} обеспечивает высокую точность замеров (до микросекунд); асинхронное подключение через SQLAlchemy engine позволяет выполнять запросы без блокировки потока; результат включает не только время выполнения, но и количество обработанных строк и текст ошибки при её возникновении.

Метод \texttt{run\_single\_test} реализует цикл выполнения итераций с потоковой отправкой метрик. На каждой итерации выполняется запрос, результаты накапливаются в буфере, и при достижении интервала в одну секунду (\texttt{\_streaming\_interval~=~1.0}) вычисляются агрегированные метрики (среднее время отклика, TPS, количество успешных и ошибочных запросов) и передаются через callback-механизм для трансляции в WebSocket и сохранения в базу данных истории.

Для сбора системных метрик используется метод \texttt{get\_system\_metrics}, который через библиотеку \texttt{psutil} получает загрузку центрального процессора, использование оперативной памяти (в мегабайтах и процентах), интенсивность дискового ввода-вывода и сетевой трафик. Метод \texttt{get\_dbms\_metrics} получает внутренние метрики СУБД через системные представления: \texttt{pg\_stat\_database} для PostgreSQL и \texttt{INFORMATION\_SCHEMA} для MySQL, включая коэффициент попаданий в кэш, количество ожиданий блокировок и взаимных блокировок.

\subsubsection*{Callback-механизм потоковой передачи данных}

Для передачи метрик из модуля генерации нагрузки в WebSocket и базу данных истории реализован паттерн Observer. Класс \texttt{TestStreamingCallback} файла \texttt{websocket\_manager.py} выступает в роли наблюдателя: он получает метрики от \texttt{LoadTester} и транслирует их подписчикам. Ниже приведён фрагмент метода \texttt{on\_metrics}:

Метод формирует объект \texttt{TestMetricsUpdate}, содержащий все метрики производительности, системные показатели и прогресс выполнения. Данные передаются через \texttt{ConnectionManager} для рассылки всем WebSocket-подписчикам теста и одновременно сохраняются в базе данных истории как точка временного ряда. Для оптимизации записи сырых выборок метрик (\texttt{MetricSample}) используется буферизация: накопление происходит до достижения размера буфера в 10 записей, после чего выполняется пакетная вставка.

\subsubsection*{Управление WebSocket-соединениями}

Управление WebSocket-соединениями реализовано в классе \texttt{ConnectionManager} файла \texttt{websocket\_manager.py}. Класс хранит активные соединения в словаре \texttt{active\_connections}, ключом которого является идентификатор теста, а значением~--- список WebSocket-объектов. Метод \texttt{connect} принимает новое соединение и добавляет его в список, метод \texttt{disconnect} удаляет соединение и очищает пустые списки. Метод \texttt{broadcast\_to\_test} рассылает JSON-сообщение всем подписчикам конкретного теста. WebSocket-эндпоинт зарегистрирован по пути \texttt{/ws/test/\{test\_id\}} и обрабатывает входящие ping-сообщения от клиента для поддержания соединения.

\subsubsection*{Управление состоянием базы данных}

Оркестратор управления состоянием баз данных реализован в классе \texttt{DatabaseStateManager} файла \texttt{database/state\_manager.py} объёмом более 400 строк. Класс управляет жизненным циклом операций: анализ запросов, создание резервной копии, выполнение теста, восстановление данных, верификация целостности.

Метод \texttt{prepare\_for\_test} выполняет подготовку базы данных к тестированию:

\begin{figure}
    \begin{minipage}{\linewidth}
\begin{lstlisting}[language=Python]
async def prepare_for_test(self, engine, dbms_type,
                           queries) -> PrepareResult:
    """Подготовка к тесту: анализ, оценка, создание backup"""
    if not self.needs_restore(queries):
        return PrepareResult(
            needs_backup=False,
            affected_tables=set(),
            size_estimate=None
        )
    
    affected_tables = self.get_affected_tables(queries)
    all_tables = await self._get_all_tables(engine, dbms_type)
    size_estimate = await self._strategy.estimate_size(
        engine, all_tables)
    backup_info = await self._strategy.create_backup(
        engine, all_tables)
    pre_fingerprint = await self._verifier.capture_fingerprint(
        engine, all_tables)
    
    return PrepareResult(
        needs_backup=True,
        affected_tables=affected_tables,
        size_estimate=size_estimate,
        backup_info=backup_info,
        pre_test_fingerprint=pre_fingerprint,
    )
\end{lstlisting}

Метод анализирует SQL-запросы через \texttt{QueryAnalyzer} на наличие write-операций (INSERT, UPDATE, DELETE). Если write-операции обнаружены, создаётся резервная копия всех таблиц базы данных через настроенную стратегию. Перед выполнением теста фиксируется «отпечаток» состояния базы данных (количество строк в каждой таблице) для последующей верификации.

Метод \texttt{restore\_after\_test} выполняет восстановление базы данных после тестирования:

\begin{figure}
    \begin{minipage}{\linewidth}
\begin{lstlisting}[language=Python]
async def restore_after_test(self, engine, dbms_type,
                             prepare_result) -> RestoreResult:
    """Восстановление данных после теста"""
    if not prepare_result.needs_backup:
        return RestoreResult(
            success=True, duration_ms=0, verified=True)
    
    start_time = time.time()
    errors: List[str] = []
    
    async with self._get_lock(dbms_type):
        try:
            await self._strategy.restore_backup(
                engine, prepare_result.backup_info)
            
            if self.config.get("verify_after_restore", True):
                post_fingerprint = await self._verifier.capture_fingerprint(
                    engine, prepare_result.affected_tables)
                verify_result = await self._verifier.verify(
                    prepare_result.pre_test_fingerprint,
                    post_fingerprint)
            
            await self._strategy.cleanup(
                engine, prepare_result.backup_info)
            
            return RestoreResult(
                success=verified,
                duration_ms=duration_ms,
                verified=verified,
                errors=errors)
        except Exception as e:
            return RestoreResult(
                success=False, errors=[str(e)])
\end{lstlisting}
    \end{minipage}
    \caption{Метод restoreAfterTest класса DatabaseStateManager}
    \label{lst:restore_after_test}
\end{figure}

Восстановление выполняется под блокировкой, специфичной для типа СУБД, что обеспечивает корректную работу при параллельном тестировании. После восстановления выполняется верификация: сравнение «отпечатка» после теста с «отпечатком» до теста. Если количество строк во всех таблицах совпадает, восстановление считается успешным. Резервные таблицы удаляются после успешного восстановления.

Стратегия резервного копирования \texttt{SqlBackupStrategy} реализует создание копий через SQL-операторы \texttt{CREATE TABLE ... AS SELECT} с префиксом \texttt{\_loadtest\_backup\_}. Восстановление выполняется через \texttt{DELETE} из оригинальных таблиц и \texttt{INSERT} из резервных копий. Такой подход обеспечивает кроссплатформенную совместимость (MySQL и PostgreSQL) без зависимости от внешних утилит командной строки.

\subsubsection*{Разрешение bundle-вариантов и запуск теста}

Фоновая задача \texttt{run\_test\_with\_streaming} создаёт экземпляр \texttt{LoadTester}, устанавливает callback для потоковой передачи метрик и callback статуса резервного копирования, после чего разрешает bundle-вариант сценария через класс \texttt{ScenarioBundleResolver}. Разрешение включает определение конкретного bundle-варианта по идентификатору bundle или шаблону сценария, извлечение SQL-запросов, параметров и спецификации индексов. Определённый bundle передаётся в \texttt{LoadTester} вместе со списком идентификаторов подключений. Перед началом теста \texttt{IndexManager} создаёт индексы, описанные в \texttt{ScenarioBundleIndex}; по завершении теста индексы удаляются. По завершении результаты собираются, сохраняются в базе данных, а статус обновляется на \texttt{completed} или \texttt{failed}. При передаче параметра \texttt{logical\_database\_id} тестовый прогон автоматически привязывается к соответствующей логической базе данных.

\subsubsection*{Модуль сравнительного анализа}

Модуль сравнительного анализа реализован в классе \texttt{ComparisonService} файла \texttt{comparison/service.py}. Конструктор класса принимает три репозитория: \texttt{repository} для загрузки данных тестов, \texttt{scenario\_bundle\_repository} для получения информации о bundle-вариантах и \texttt{connection\_repository} для получения данных о подключениях. Основной метод \texttt{analyze} принимает список идентификаторов тестов, идентификатор базового теста и конфигурацию отчёта, загружает данные из репозиториев, определяет тип сравнения и выполняет попарное сопоставление метрик:

async def analyze(self, test_ids, baseline_id=None,
                  report_config=None):
    """Выполнить сравнительный анализ по выбранным тестам"""
    unique_test_ids = self._normalize_test_ids(test_ids)
    baseline_uuid = self._resolve_baseline_id(

Метод \texttt{\_detect\_comparison\_type} автоматически определяет тип сравнения на основе сигнатур нагрузки: \texttt{CROSS\_DATABASE} (одинаковая нагрузка, разные СУБД), \texttt{SCALABILITY} (одна СУБД, разное число потоков), \texttt{TEMPORAL} (одинаковая конфигурация, разное время выполнения) или \texttt{MIXED} (смешанный сценарий). Для каждого типа сравнения формируются соответствующие предупреждения о сопоставимости результатов.

Сбор сырых выборок метрик реализован с механизмом fallback: при наличии записей \texttt{MetricSample} используются они, при их отсутствии~--- данные из \texttt{TimeSeries}, и только в крайнем случае~--- агрегированные метрики из \texttt{TestResult} с соответствующим предупреждением. Это обеспечивает работоспособность сравнения даже для тестов, выполненных до добавления модуля сбора сырых метрик.

\subsubsection*{Реализация статистических функций}

Модуль \texttt{comparison/statistics.py} содержит математический аппарат для статистического сравнения выборок. Функция \texttt{calculate\_descriptive\_stats} вычисляет описательную статистику через \texttt{numpy}: количество, среднее, медиану, стандартное отклонение, минимум, максимум, процентили p50, p95, p99. Функция \texttt{check\_normality} проверяет нормальность распределения через критерий Шапиро-Уилка (\texttt{scipy.stats.shapiro}) с ограничением выборки в 5000 элементов.

Функция \texttt{compare\_two\_samples} выполняет попарное сравнение двух выборок: при нормальном распределении применяется критерий Уэлча (\texttt{scipy.stats.ttest\_ind} с \texttt{equal\_var=False}), при ненормальном~--- критерий Манна-Уитни (\texttt{scipy.stats.mannwhitneyu} с \texttt{alternative="two-sided"}). Минимальный размер выборки для статистического теста установлен в 10 элементов, уровень значимости~--- 0,05. Результат включает процентное отклонение, использованный тест, статистику, p-значение и текстовую интерпретацию.

\subsubsection*{Обработка ошибок}

Обработка ошибок в серверной части реализована на нескольких уровнях. На уровне API маршрутов все исключения перехватываются и преобразуются в HTTP-статусы: \texttt{ValueError}~--- в 400 Bad Request, неожиданные исключения~--- в 500 Internal Server Error. На уровне бизнес-логики (\texttt{LoadTester}, \texttt{DatabaseStateManager}) ошибки логируются с указанием контекста (тип СУБД, идентификатор запроса) и, где возможно, обрабатываются без прерывания выполнения теста. Блок \texttt{try/finally} в методах тестирования гарантирует выполнение восстановления базы данных даже при критических ошибках.

При возникновении ошибки выполнения SQL-запроса метод \texttt{execute\_query} не прерывает цикл итераций, а сохраняет текст ошибки в результате и продолжает выполнение следующих запросов. Это позволяет получить полную картину производительности, включая частоту ошибок, которая сама по себе является важной метрикой.

\subsubsection*{Слой доступа к данным}

Слой доступа к данным реализован через паттерн Repository. Базовый класс \texttt{BaseRepository} обеспечивает асинхронное подключение к базе данных истории через SQLAlchemy с управлением сессиями через контекстный менеджер. Наследники реализуют специфичные операции для своих сущностей: \texttt{TestRepository} управляет тестовыми прогонами и временными рядами, \texttt{ConnectionRepository}~--- конфигурациями подключений с расшифровкой паролей, \texttt{ProfileRepository}~--- профилями схем и шаблонами сценариев, \texttt{ScenarioBundleRepository}~--- bundle-вариантами с их запросами и индексами, \texttt{LogicalDatabaseRepository}~--- логическими базами данных.

\subsubsection*{Управление подключениями к СУБД}

Класс \texttt{DatabaseConnection} файла \texttt{database/connection.py} управляет подключениями к тестируемым СУБД. Подключения хранятся в репозитории базы данных истории и загружаются лениво при первом асинхронном вызове. Реализация ленивой инициализации \texttt{AsyncEngine} представлена ниже:

\begin{figure}
    \begin{minipage}{\linewidth}
\begin{lstlisting}[language=Python]
async def get_engine_async(self, connection_key):
    """Получение engine с загрузкой подключений из БД"""
    await self.ensure_connection_config(connection_key)
    return self.get_engine(connection_key)

def get_engine(self, connection_key) -> AsyncEngine:
    """Получение или создание async engine"""
    connection_key = self._resolve_connection_key(connection_key)
    if connection_key not in self.engines:
        connection_string = self.get_connection_string(
            connection_key)
        self.engines[connection_key] = create_async_engine(
            connection_string,
            pool_size=5,
            max_overflow=10,
            echo=False
        )
    return self.engines[connection_key]
\end{lstlisting}
    \end{minipage}
    \caption{Ленивая инициализация AsyncEngine}
    \label{lst:lazy_engine}
\end{figure}

Метод \texttt{\_resolve\_connection\_key} обеспечивает поддержку shorthand-ключей: если передан ключ \texttt{mysql} или \texttt{postgresql} и существует ровно одно активное подключение данного типа, оно используется автоматически. При наличии нескольких подключений одного типа требуется явное указание \texttt{connection\_id}. Кэширование конфигураций подключений в памяти (\texttt{\_connection\_configs}) исключает повторные обращения к базе данных при каждом запросе.

Для шифрования паролей подключений используется модуль \texttt{core/crypto.py}, реализующий симметричное шифрование AES-256. Пароль шифруется перед сохранением в базе данных и расшифровывается при установлении подключения к тестируемой СУБД. Ключ шифрования задаётся через переменную окружения \texttt{ENCRYPTION\_KEY}.

\subsection{Реализация клиентской части}

Клиентская часть системы реализована как одностраничное приложение на базе Next.js 16 с клиентской маршрутизацией. Главная страница (\texttt{app/page.tsx}) определяет компонент-маршрутизатор, который условно отображает один из девяти компонентов-страниц в зависимости от значения \texttt{currentPage} в Zustand store. Общая компоновка включает верхнюю панель (\texttt{Header}) с логотипом и навигацией, боковое меню (\texttt{Sidebar}) с переключением страниц и основную область контента.

\subsubsection*{Архитектура компонентов}

Страница \texttt{HomePage} является стартовой и предоставляет краткое описание системы с кнопками быстрого перехода к конфигурации теста или истории. \texttt{ConnectionsPage} обеспечивает полный CRUD для сохранённых подключений к СУБД: добавление, редактирование, удаление, проверка доступности. При создании подключения страница отображает результат автоопределения профиля схемы с коэффициентом уверенности.

Страница \texttt{ConfigPage} является центральным узлом конфигурации теста. Она содержит несколько карточек: выбор логической базы данных или отдельных подключений (\texttt{LogicalDbSelectorCard} и \texttt{DatabaseSelectionCard}), выбор режима теста (\texttt{TestModeSelectorCard}), выбор сценария или bundle-варианта (\texttt{ScenarioSelectorCard}), настройка параметров нагрузки: числа виртуальных пользователей, числа итераций и времени прогрева (\texttt{SliderConfigCard}), а также сводка конфигурации и кнопка запуска (\texttt{ConfigSummaryCard}).

Страница \texttt{ScenariosPage} предоставляет браузер bundle-вариантов через компонент \texttt{logical-scenarios-browser}. В браузере отображаются профили схем и связанные с ними bundle'ы с их SQL-запросами, параметрами и спецификацией индексов. Предусмотрены операции создания, редактирования, клонирования и генерации bundle'ов из эталонного подключения.

Страница \texttt{DashboardsPage} предоставляет дашборд реального времени с четырьмя вкладками: метрики транзакций (\texttt{TransactionMetricsTab}), метрики базы данных (\texttt{DatabaseMetricsTab}), метрики СУБД (\texttt{DbmsMetricsTab}) и системные метрики (\texttt{SystemMetricsTab}). Каждая вкладка отображает временны\'е ряды соответствующих метрик для всех подключённых баз данных одновременно.

Страница \texttt{HistoryPage} отображает таблицу выполненных тестов с возможностью выбора от 2 до 5 тестов для сравнения. Страница \texttt{ComparisonPage} отображает результаты сравнительного анализа: таблицу метрик с процентными отклонениями от базового теста, диаграммы, результаты статистических тестов и аналитический отчёт. Страница \texttt{DatabaseStatePage} предоставляет панель управления состоянием подключённых баз данных~--- ручное создание и восстановление резервных копий. Страница \texttt{ReportsPage} обеспечивает формирование и экспорт отчётов сравнительного анализа в формате JSON.

\subsubsection*{Управление состоянием (Zustand)}

Состояние приложения хранится в едином store, реализованном через библиотеку Zustand. Файл \texttt{lib/store.ts} определяет интерфейс \texttt{AppState} и создаёт store через \texttt{create<AppState>}. Store включает следующие группы данных: навигация (текущая страница, состояние боковой панели), конфигурация теста (режим, сценарий, идентификатор bundle-варианта, признак использования индексов, число виртуальных пользователей, итерации, время прогрева), текущий тест, история тестов, данные реального времени (временны\'е ряды для каждого подключения), словари имён и типов подключений, параметры сравнительного анализа.

Ключом в словаре \texttt{realtimeData} выступает не тип СУБД, а уникальный идентификатор подключения (\texttt{connection\_key}), что позволяет отображать метрики для нескольких экземпляров одного типа СУБД независимо.

\subsubsection*{Реализация WebSocket-хука}

Хук \texttt{useTestWebSocket} файла \texttt{hooks/use-test-websocket.ts} обеспечивает двустороннюю связь с сервером для получения метрик в реальном времени. При монтировании компонента с валидным \texttt{testId} автоматически устанавливается WebSocket-соединение по адресу \texttt{/ws/test/\{testId\}}. Каждые 30 секунд отправляется ping-сообщение для предотвращения разрыва соединения из-за таймаута.

Обработка входящих сообщений реализована через \texttt{handleMessage}:

Переподключение выполняется только при статусе \texttt{running}, что предотвращает бесконечные попытки подключения после завершения теста. При получении сообщения типа \texttt{metrics} данные преобразуются в \texttt{TimeSeriesPoint} и добавляются в store через \texttt{addRealtimeData}, что вызывает реактивное обновление компонентов дашборда.

\subsubsection*{Модульный API-клиент}

API-клиент реализован как класс \texttt{ApiClient} файла \texttt{lib/api/client.ts}. Базовый URL определяется переменной окружения \texttt{NEXT\_PUBLIC\_API\_URL}. Метод \texttt{request<T>} централизует обработку ошибок: при получении HTTP-статуса, отличного от 2xx, из ответа извлекается сообщение об ошибке и выбрасывается исключение \texttt{Error}. Все методы клиента типизированы через дженерики TypeScript, что обеспечивает проверку корректности данных на этапе компиляции.

Класс включает доменные методы для каждой группы эндпоинтов: методы управления подключениями и их профилями (\texttt{getConnections}, \texttt{assignConnectionProfile}), методы работы с профилями схемы (\texttt{getSchemaProfiles}, \texttt{generateBundles}), методы управления логическими базами данных (\texttt{getLogicalDatabases}, \texttt{assignLogicalDatabaseProfile}), а также методы запуска тестов (\texttt{runAsyncTest}), получения истории (\texttt{getHistoryTests}) и запуска сравнительного анализа (\texttt{analyzeComparison}). Экземпляр \texttt{apiClient} экспортируется как singleton, что обеспечивает единый базовый URL и централизованную обработку ошибок для всех компонентов приложения.

\subsubsection*{Визуализация данных}

Визуализация метрик реализована через библиотеку Recharts. На странице дашборда отображаются графики времени отклика, пропускной способности, загрузки CPU и использования RAM в реальном времени; каждый компонент временно\'го ряда реализован в переиспользуемом компоненте \texttt{TimeSeriesChart}. На странице сравнительного анализа отображаются таблица метрик с процентными отклонениями от базового теста, результаты статистических тестов с p-значениями и интерпретациями, графики сравнения (bar chart, box plot, throughput over time), а также секции аналитического отчёта, конфигурируемые пользователем.

Стилизация интерфейса выполнена через Tailwind CSS v4 с использованием дизайн-токенов для цветов, отступов и размеров шрифтов. Поддерживается тёмная тема. Компоненты интерфейса построены на базе Radix UI через библиотеку shadcn/ui, обеспечивающей доступность (accessibility) и корректную работу с клавиатурой. Уведомления пользователю отображаются через библиотеку Sonner в виде всплывающих сообщений.

\subsection{Контейнеризация и развёртывание}

Развёртывание системы организовано через Docker Compose. Файл \texttt{docker-compose.yml} определяет три сервиса, охватывающих всю систему, и представлен в таблице~\ref{tab:docker_services}.

\begin{table}
    \centering
    \caption{Сервисы Docker Compose}
    \label{tab:docker_services}
    \begin{tabularx}{\textwidth}{l l X l}
        \hline
        Сервис & Образ/источник & Назначение & Порт \\
        \hline
        project-db & postgres:15-alpine & База данных проекта (история, сценарии, профили) & 5433:5432 \\
        backend & Dockerfile (Python) & FastAPI REST API и WebSocket-сервер & 8000:8000 \\
        frontend & Dockerfile (Node.js) & Next.js веб-интерфейс & 3000:3000 \\
        \hline
    \end{tabularx}
\end{table}

Сервис \texttt{project-db} развёртывается на базе образа \texttt{postgres:15-alpine}. Сервис настроен с именованным томом \texttt{project\_db\_data} для персистентности данных, healthcheck через \texttt{pg\_isready} с интервалом 5 секунд и политикой перезапуска \texttt{unless-stopped}. Порт 5433 используется для обеспечения совместимости с окружениями, где порт 5432 уже занят.

Сервис \texttt{backend} собирается из \texttt{backend/Dockerfile} и запускает FastAPI-приложение через uvicorn. Зависимость \texttt{depends\_on} с условием \texttt{service\_healthy} гарантирует, что бэкенд не стартует до полной готовности базы данных. Параметр \texttt{extra\_hosts: host.docker.internal:host-gateway} обеспечивает доступность хостовых СУБД (Sakila/Pagila) из контейнера по имени \texttt{host.docker.internal}.

Сервис \texttt{frontend} собирается из \texttt{frontend/Dockerfile} и запускает Next.js-приложение. Переменные \texttt{NEXT\_PUBLIC\_API\_URL} и \texttt{NEXT\_PUBLIC\_WS\_URL} задаются на этапе сборки и указывают на бэкенд-сервис. Все три сервиса объединены в пользовательскую сеть \texttt{loadtest-network} типа bridge, обеспечивающую изолированное сетевое взаимодействие между компонентами.

Тестовые СУБД развёртываются отдельно через собственные Docker Compose-файлы в директориях \texttt{sakila/} и \texttt{pagila/}. MySQL с схемой Sakila и PostgreSQL с схемой Pagila при первом запуске автоматически инициализируют схему через скрипты инициализации соответствующих образов. Оба контейнера доступны из бэкенда через хост \texttt{host.docker.internal}.

\subsubsection*{Инициализация и миграции}

Инициализация схемы базы данных проекта полностью автоматизирована и не требует ручных действий. При каждом старте бэкенда \texttt{main.py} через менеджер контекста lifespan выполняет команду Alembic \texttt{upgrade head}, применяющую все ожидающие миграции. На момент разработки схема включает 10 версионируемых миграций: создание базовой схемы, заполнение встроенных сценариев, добавление индексов сценариев, поддержка привязки подключений, добавление профилей и bundle'ов, удаление устаревших встроенных профилей, добавление поддержки логических баз данных, привязка профилей к логическим базам данных, очистка, добавление поля \texttt{logical\_database\_id} в тестовые прогоны.

После применения миграций запускается \texttt{LogicalScenarioBootstrap}, который создаёт встроенные шаблоны сценариев (\texttt{mixed\_light}, \texttt{read\_only}, \texttt{mixed\_heavy} и другие) и встроенные профили схем, соответствующие схемам Sakila и Pagila. При наличии зарегистрированных подключений с определёнными профилями bootstrap автоматически генерирует bundle-варианты для всех пар <<профиль + шаблон>>. Описанная процедура полностью идемпотентна: при повторном запуске уже созданные сущности не дублируются.

Конфигурация системы задаётся через файл \texttt{.env}: переменная \texttt{HISTORY\_DATABASE\_URL} задаёт подключение к базе данных проекта, \texttt{ENCRYPTION\_KEY}~--- ключ шифрования паролей подключений, \texttt{API\_HOST} и \texttt{API\_PORT}~--- настройки API-сервера, \texttt{NEXT\_PUBLIC\_WS\_URL} и \texttt{NEXT\_PUBLIC\_API\_URL}~--- адреса для фронтенда. Файл \texttt{.env} не включается в систему контроля версий; вместо него поставляется шаблон \texttt{env.example}.

Полный запуск системы сводится к одной команде: \texttt{docker compose up -d --build}. После её выполнения все три сервиса (база данных, бэкенд, фронтенд) будут запущены, миграции применены автоматически, а bootstrap инициализирует встроенные сценарии и профили. Веб-интерфейс становится доступен по адресу \texttt{http://localhost:3000}, документация API~--- по адресу \texttt{http://localhost:8000/docs}.
