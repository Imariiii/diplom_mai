\section{ПРОГРАММНАЯ РЕАЛИЗАЦИЯ}

В данной главе описывается процесс программной реализации разрабатываемой системы сравнительного нагрузочного тестирования СУБД. Обосновывается выбор технологического стека, рассматривается реализация серверной и клиентской частей, а также вопросы контейнеризации и развёртывания.

\subsection{Выбор и обоснование технологического стека}

Выбор технологического стека определялся требованиями к производительности, поддерживаемости и удобству разработки. Для серверной части был выбран язык Python версии 3.9 и выше в сочетании с фреймворком FastAPI. Python обеспечивает высокую скорость разработки благодаря развитой экосистеме библиотек для работы с базами данных, статистического анализа и визуализации. FastAPI предоставляет современный асинхронный веб-фреймворк с автоматической генерацией документации API через OpenAPI, встроенной валидацией данных через Pydantic и высокой производительностью, сопоставимой с фреймворками на Node.js.

Для асинхронного выполнения запросов к СУБД выбраны драйверы asyncpg для PostgreSQL и aiomysql для MySQL. Оба драйвера обеспечивают неблокирующий ввод-вывод, что позволяет эффективно обрабатывать множество параллельных соединений в рамках одного потока операционной системы. В качестве ORM-слоя используется SQLAlchemy версии 2.0 с поддержкой асинхронных сессий, что обеспечивает типобезопасную работу с базой данных истории и абстрагирует SQL-запросы от бизнес-логики.

Для клиентской части выбран фреймворк Next.js версии 16 на базе React 19 с использованием TypeScript. Next.js обеспечивает серверный рендеринг, оптимизацию маршрутизации и встроенную поддержку API-прокси. TypeScript гарантирует типобезопасность на всех уровнях клиентского приложения, снижая количество ошибок времени выполнения. Для управления состоянием приложения выбрана библиотека Zustand, отличающаяся минимальным объёмом кода и отсутствием необходимости в оборачивании приложения в провайдеры. Для визуализации данных используется библиотека Recharts, предоставляющая декларативный API для построения графиков на базе SVG. Стилизация реализована через Tailwind CSS версии 4 с использованием компонентной библиотеки Radix UI.

Для развёртывания базы данных истории используется Docker с образом PostgreSQL 15 Alpine, что обеспечивает воспроизводимость окружения и упрощает развёртывание на различных платформах. Тестовые СУБД (MySQL и PostgreSQL) также развёртываются в Docker-контейнерах с использованием образов с предустановленными схемами Sakila и Pagila.

\subsection{Реализация серверной части}

Серверная часть системы реализована в виде модульного приложения FastAPI. Точка входа находится в файле \texttt{main.py}, где инициализируется экземпляр приложения, настраивается CORS-мидлвэр для разрешения кросс-доменных запросов от фронтенда, регистрируются маршруты и создаётся WebSocket-эндпоинт.

Инициализация компонентов системы выполняется через модуль \texttt{initialize.py}, который создаёт экземпляры репозиториев (\texttt{TestRepository}, \texttt{ScenarioRepository}, \texttt{ConnectionRepository}) и передаёт их зависимым компонентам. Такой подход реализует паттерн внедрения зависимостей (dependency injection) и обеспечивает централизованное управление жизненным циклом компонентов.

\subsubsection*{Модуль генерации нагрузки}

Модуль генерации нагрузки реализован в классе \texttt{LoadTester} файла \texttt{load\_tester/tester.py} объёмом более 1400 строк. Класс содержит методы для выполнения отдельных запросов, запуска одиночного теста, сравнительного теста для нескольких СУБД и полного набора тестов по всем запросам с прогревом. Каждый метод следует единой структуре: подготовка базы данных, выполнение нагрузки, сбор метрик, восстановление состояния, верификация. Блок \texttt{try/finally} гарантирует выполнение восстановления даже при возникновении ошибок в ходе тестирования.

Базовый метод \texttt{execute\_query} выполняет отдельный SQL-запрос к указанной базе данных, замеряя время отклика через \texttt{time.perf\_counter()}. Ниже приведён фрагмент реализации:

\begin{figure}
    \begin{minipage}{\linewidth}
\begin{lstlisting}[language=Python]
async def execute_query(self, db_key, query, query_id):
    """Выполнение одного запроса с измерением времени"""
    start_time = time.perf_counter()
    error = None
    rows_count = 0
    db_type = self.db_connection.get_dbms_type(db_key)
    
    try:
        engine = await self.db_connection.get_engine_async(db_key)
        async with engine.connect() as conn:
            result = await conn.execute(text(query))
            rows_count = len(result.fetchall()) if result.returns_rows else 0
            await conn.commit()
    except Exception as e:
        error = str(e)
    
    end_time = time.perf_counter()
    execution_time = (end_time - start_time) * 1000
    
    return {
        'query_id': query_id,
        'execution_time_ms': execution_time,
        'rows_count': rows_count,
        'error': error,
        'timestamp': datetime.now(timezone.utc).isoformat()
    }
\end{lstlisting}
    \end{minipage}
    \caption{Реализация метода executeQuery класса LoadTester}
    \label{lst:execute_query}
\end{figure}

Ключевые особенности реализации: использование \texttt{time.perf\_counter()} обеспечивает высокую точность замеров (до микросекунд); асинхронное подключение через SQLAlchemy engine позволяет выполнять запросы без блокировки потока; результат включает не только время выполнения, но и количество обработанных строк и текст ошибки при её возникновении.

Метод \texttt{run\_full\_test\_suite} реализует запуск полного набора тестов по всем запросам для указанных СУБД. Ниже приведён фрагмент, демонстрирующий организацию цикла тестирования:

\begin{figure}
    \begin{minipage}{\linewidth}
\begin{lstlisting}[language=Python]
async def run_full_test_suite(
    self,
    db_types: List[str] = None,
    iterations: int = 10,
    duration: int = 60,
    virtual_users: int = 10,
    scenario: str = "mixed_light",
    warmup_time: int = 5
) -> List[Dict]:
    """Запуск полного набора тестов"""
    if db_types is None:
        db_types = ['mysql', 'postgresql']
    
    queries = self.query_manager.get_all_queries()
    all_results = []
    total_queries = len(queries)
    
    if self._metrics_callback:
        self._metrics_callback.set_total_queries(total_queries)
        await self._metrics_callback.on_test_start()
    
    # Прогрев (если указан)
    if warmup_time > 0:
        print(f"Прогрев системы ({warmup_time} сек)...")
        await self._emit_status("running",
            f"Прогрев системы ({warmup_time} сек)...")
        warmup_query = queries[0] if queries else None
        if warmup_query:
            for db_type in db_types:
                for _ in range(min(5, iterations)):
                    await self.execute_query(
                        db_type, warmup_query['sql'],
                        warmup_query['id'])
                    await asyncio.sleep(0.1)
        await asyncio.sleep(warmup_time)
    
    for idx, query in enumerate(queries):
        comparison = await self.run_comparison_test(
            query['id'], db_types, iterations,
            virtual_users=virtual_users, scenario=scenario)
        all_results.append(comparison)
    
    return all_results
\end{lstlisting}
    \end{minipage}
    \caption{Метод runFullTestSuite класса LoadTester}
    \label{lst:run_full_test_suite}
\end{figure}

Метод реализует фазу прогрева, в которой каждый запрос выполняется минимальное количество раз для заполнения кэшей СУБД, после чего следует основной цикл тестирования. Для каждого запроса вызывается метод \texttt{run\_comparison\_test}, который последовательно тестирует все указанные СУБД. Прогресс выполнения передаётся через callback-механизм для отображения в пользовательском интерфейсе.

Метод \texttt{run\_single\_test} реализует цикл выполнения итераций с потоковой отправкой метрик. На каждой итерации выполняется запрос, результаты накапливаются в буфере, и при достижении интервала в одну секунду (\texttt{\_streaming\_interval = 1.0}) вычисляются агрегированные метрики (среднее время отклика, TPS, количество успешных и ошибочных запросов) и передаются через callback-механизм для трансляции в WebSocket и сохранения в базу данных истории.

Для сбора системных метрик используется метод \texttt{get\_system\_metrics}, который через библиотеку \texttt{psutil} получает загрузку центрального процессора, использование оперативной памяти (в мегабайтах и процентах), интенсивность дискового ввода-вывода (IOPS) и сетевой трафик. Метод \texttt{get\_dbms\_metrics} получает внутренние метрики СУБД через системные представления: \texttt{pg\_stat\_database} для PostgreSQL и \texttt{INFORMATION\_SCHEMA} для MySQL, включая коэффициент попаданий в кэш, количество ожиданий блокировок и взаимных блокировок.

\subsubsection*{Управление WebSocket-соединениями}

Управление WebSocket-соединениями реализовано в классе \texttt{ConnectionManager} файла \texttt{websocket\_manager.py}. Класс хранит активные соединения в словаре \texttt{active\_connections}, ключом которого является идентификатор теста, а значением — список WebSocket-объектов. Метод \texttt{connect} принимает новое соединение и добавляет его в список, метод \texttt{disconnect} удаляет соединение и очищает пустые списки. Метод \texttt{broadcast\_to\_test} рассылает JSON-сообщение всем подписчикам конкретного теста.

Класс \texttt{TestStreamingCallback} связывает \texttt{LoadTester} с \texttt{ConnectionManager}: при получении метрик от \texttt{LoadTester} callback отправляет их через WebSocket и сохраняет в базе данных истории. Для оптимизации производительности записи метрик \texttt{MetricSample} буферизуются: накопление происходит до достижения размера буфера в 10 записей, после чего выполняется пакетная вставка. Это снижает нагрузку на базу данных истории при высокой интенсивности тестирования.

\subsubsection*{Управление состоянием базы данных}

Оркестратор управления состоянием баз данных реализован в классе \texttt{DatabaseStateManager} файла \texttt{database/state\_manager.py} объёмом более 400 строк. Класс управляет жизненным циклом: анализ запросов, создание резервной копии, выполнение теста, восстановление данных, верификация целостности.

Метод \texttt{prepare\_for\_test} выполняет подготовку базы данных к тестированию. Ниже приведён фрагмент реализации:

\begin{figure}
    \begin{minipage}{\linewidth}
\begin{lstlisting}[language=Python]
async def prepare_for_test(self, engine, dbms_type,
                           queries) -> PrepareResult:
    """Подготовка к тесту: анализ, оценка, создание backup"""
    # Проверяем, нужен ли backup
    if not self.needs_restore(queries):
        return PrepareResult(
            needs_backup=False,
            affected_tables=set(),
            size_estimate=None
        )
    
    # Получаем затронутые таблицы
    affected_tables = self.get_affected_tables(queries)
    
    # Получаем ВСЕ таблицы базы для полного backup
    all_tables = await self._get_all_tables(engine, dbms_type)
    
    # Оцениваем размер для всех таблиц
    size_estimate = await self._strategy.estimate_size(
        engine, all_tables)
    
    # Создаём backup для ВСЕХ таблиц
    backup_info = await self._strategy.create_backup(
        engine, all_tables)
    
    # Сохраняем фингерпринт до теста
    pre_fingerprint = await self._verifier.capture_fingerprint(
        engine, all_tables)
    
    return PrepareResult(
        needs_backup=True,
        affected_tables=affected_tables,
        size_estimate=size_estimate,
        backup_info=backup_info,
        pre_test_fingerprint=pre_fingerprint,
    )
\end{lstlisting}
    \end{minipage}
    \caption{Метод prepareForTest класса DatabaseStateManager}
    \label{lst:prepare_for_test}
\end{figure}

Метод анализирует SQL-запросы через \texttt{QueryAnalyzer} на наличие write-операций (INSERT, UPDATE, DELETE). Если write-операции обнаружены, создаётся резервная копия всех таблиц базы данных через стратегию \texttt{SqlBackupStrategy}. Перед выполнением теста фиксируется «отпечаток» состояния базы данных (количество строк в каждой таблице) для последующей верификации.

Метод \texttt{restore\_after\_test} выполняет восстановление базы данных после тестирования:

\begin{figure}
    \begin{minipage}{\linewidth}
\begin{lstlisting}[language=Python]
async def restore_after_test(self, engine, dbms_type,
                             prepare_result) -> RestoreResult:
    """Восстановление данных после теста"""
    if not prepare_result.needs_backup:
        return RestoreResult(
            success=True, duration_ms=0, verified=True)
    
    start_time = time.time()
    errors: List[str] = []
    
    async with self._get_lock(dbms_type):
        try:
            # Выполняем restore
            await self._strategy.restore_backup(
                engine, prepare_result.backup_info)
            
            # Верификация
            if self.config.get("verify_after_restore", True):
                post_fingerprint = await self._verifier.capture_fingerprint(
                    engine, prepare_result.affected_tables)
                verify_result = await self._verifier.verify(
                    prepare_result.pre_test_fingerprint,
                    post_fingerprint)
            
            # Очищаем backup после restore
            await self._strategy.cleanup(
                engine, prepare_result.backup_info)
            
            return RestoreResult(
                success=verified,
                duration_ms=duration_ms,
                verified=verified,
                errors=errors)
        except Exception as e:
            return RestoreResult(
                success=False, errors=[str(e)])
\end{lstlisting}
    \end{minipage}
    \caption{Метод restoreAfterTest класса DatabaseStateManager}
    \label{lst:restore_after_test}
\end{figure}

Восстановление выполняется под блокировкой, специфичной для типа СУБД, что обеспечивает корректную работу при параллельном тестировании. После восстановления выполняется верификация: сравнение «отпечатка» после теста с «отпечатком» до теста. Если количество строк во всех таблицах совпадает, восстановление считается успешным. Резервные таблицы удаляются после успешного восстановления.

Стратегия резервного копирования \texttt{SqlBackupStrategy} реализует создание копий через SQL-операторы \texttt{CREATE TABLE ... AS SELECT} с префиксом \texttt{\_loadtest\_backup\_}. Восстановление выполняется через \texttt{DELETE} из оригинальных таблиц и \texttt{INSERT} из резервных копий. Такой подход обеспечивает кроссплатформенную совместимость (MySQL и PostgreSQL) без зависимости от утилит \texttt{mysqldump} и \texttt{pg\_dump}.

\subsubsection*{Модуль сравнительного анализа}

Модуль сравнительного анализа реализован в классе \texttt{ComparisonService} файла \texttt{comparison/service.py}. Основной метод \texttt{analyze} принимает список идентификаторов тестов и идентификатор базового теста, загружает данные из репозитория, определяет тип сравнения и выполняет попарное сопоставление метрик.

\begin{figure}

async def analyze(self, test_ids, baseline_id=None,

Метод \texttt{\_detect\_comparison\_type} автоматически определяет тип сравнения на основе сигнатур нагрузки: \texttt{CROSS\_DATABASE} (одинаковая нагрузка, разные СУБД), \texttt{SCALABILITY} (одна СУБД, разное число потоков), \texttt{TEMPORAL} (одинаковая конфигурация, разное время выполнения) или \texttt{MIXED} (смешанный сценарий). Для каждого типа сравнения формируются соответствующие предупреждения о сопоставимости результатов.

Сбор сырых выборок метрик реализован с механизмом fallback: при наличии записей \texttt{MetricSample} используются они, при их отсутствии — данные из \texttt{TimeSeries}, и только в крайнем случае — агрегированные метрики из \texttt{TestResult} с соответствующим предупреждением. Это обеспечивает работоспособность сравнения даже для тестов, выполненных до добавления модуля сбора сырых метрик.

\subsubsection*{Реализация статистических функций}

Модуль \texttt{comparison/statistics.py} содержит математический аппарат для статистического сравнения выборок. Функция \texttt{calculate\_descriptive\_stats} вычисляет описательную статистику через \texttt{numpy}: количество, среднее, медиану, стандартное отклонение, минимум, максимум, процентили p50, p95, p99. Функция \texttt{check\_normality} проверяет нормальность распределения через критерий Шапиро-Уилка (\texttt{scipy.stats.shapiro}) с ограничением выборки в 5000 элементов.

Функция \texttt{compare\_two\_samples} выполняет попарное сравнение двух выборок: при нормальном распределении применяется критерий Уэлча (\texttt{scipy.stats.ttest\_ind} с \texttt{equal\_var=False}), при ненормальном — критерий Манна-Уитни (\texttt{scipy.stats.mannwhitneyu} с \texttt{alternative="two-sided"}). Минимальный размер выборки для статистического теста установлен в 10 элементов, уровень значимости — 0,05. Результат включает процентное отклонение, использованный тест, статистику, p-значение и текстовую интерпретацию.

Модуль \texttt{history\_routes.py} обеспечивает доступ к истории тестов с пагинацией и фильтрацией по статусу. Модуль \texttt{comparison\_routes.py} реализует эндпоинт \texttt{POST /comparison/analyze} для запуска сравнительного анализа. Модуль \texttt{settings\_routes.py} управляет настройками восстановления базы данных.

\subsubsection*{Обработка ошибок}

Обработка ошибок в серверной части реализована на нескольких уровнях. На уровне API маршрутов все исключения перехватываются и преобразуются в HTTP-статусы: \texttt{ValueError} — в 400 Bad Request, неожиданные исключения — в 500 Internal Server Error. На уровне бизнес-логики (\texttt{LoadTester}, \texttt{DatabaseStateManager}) ошибки логируются с указанием контекста (тип СУБД, идентификатор запроса) и, где возможно, обрабатываются без прерывания выполнения теста. Блок \texttt{try/finally} в методах тестирования гарантирует выполнение восстановления базы данных даже при критических ошибках.

При возникновении ошибки выполнения SQL-запроса метод \texttt{execute\_query} не прерывает цикл итераций, а сохраняет текст ошибки в результате и продолжает выполнение следующих запросов. Это позволяет получить полную картину производительности, включая частоту ошибок, которая сама по себе является важной метрикой.

\subsubsection*{Слой доступа к данным (Repository Pattern)}

Слой доступа к данным реализован через паттерн Repository. Базовый класс \texttt{BaseRepository} обеспечивает асинхронное подключение к базе данных истории через SQLAlchemy. Наследники — \texttt{TestRepository}, \texttt{ScenarioRepository}, \texttt{ConnectionRepository} — реализуют специфичные операции для своих сущностей.

Метод \texttt{TestRepository.save\_metric\_samples} выполняет асинхронное сохранение сырых выборок метрик:

\subsubsection*{Управление подключениями к СУБД}

Класс \texttt{DatabaseConnection} файла \texttt{database/connection.py} объёмом более 230 строк управляет подключениями к тестируемым СУБД. Подключения хранятся в репозитории базы данных истории и загружаются лениво при первом асинхронном вызове.

Реализация ленивой инициализации \texttt{AsyncEngine}:

\begin{figure}
    \begin{minipage}{\linewidth}
\begin{lstlisting}[language=Python]
async def get_engine_async(self, connection_key):
    """Получение engine с загрузкой подключений из БД"""
    await self.ensure_connection_config(connection_key)
    return self.get_engine(connection_key)

def get_engine(self, connection_key) -> AsyncEngine:
    """Получение или создание async engine"""
    connection_key = self._resolve_connection_key(connection_key)
    if connection_key not in self.engines:
        connection_string = self.get_connection_string(
            connection_key)
        self.engines[connection_key] = create_async_engine(
            connection_string,
            pool_size=5,
            max_overflow=10,
            echo=False
        )
    return self.engines[connection_key]
\end{lstlisting}
    \end{minipage}
    \caption{Ленивая инициализация AsyncEngine}
    \label{lst:lazy_engine}
\end{figure}

Метод \texttt{\_resolve\_connection\_key} обеспечивает поддержку shorthand-ключей: если передан ключ \texttt{mysql} или \texttt{postgresql} и существует ровно одно активное подключение данного типа, оно используется автоматически. При наличии нескольких подключений одного типа требуется явное указание \texttt{connection\_id}. Кэширование конфигураций подключений в памяти (\texttt{\_connection\_configs}) исключает повторные обращения к базе данных при каждом запросе.

Для шифрования паролей подключений используется модуль \texttt{core/crypto.py}, реализующий симметричное шифрование AES-256. Пароль шифруется перед сохранением в базе данных и расшифровывается при установлении подключения к тестируемой СУБД. Ключ шифрования задаётся через переменную окружения \texttt{ENCRYPTION\_KEY}.

Конфигурация системы реализована через класс \texttt{Settings} на базе \texttt{pydantic-settings}. Все параметры (URL базы данных истории, настройки восстановления, пути к директориям, параметры API) загружаются из переменных окружения и файла \texttt{.env}, что соответствует принципам Twelve-Factor App.

\subsection{Реализация клиентской части}

Клиентская часть системы реализована как одностраничное приложение на базе Next.js 16 с клиентской маршрутизацией. Главная страница (\texttt{app/page.tsx}) определяет компонент-маршрутизатор, который условно отображает один из семи компонентов-страниц в зависимости от значения \texttt{currentPage} в Zustand store. Такой подход обеспечивает плавные переходы между страницами без перезагрузки браузера.

\subsubsection*{Управление состоянием (Zustand)}

Состояние приложения хранится в едином store, реализованном через библиотеку Zustand. Файл \texttt{lib/store.ts} объёмом около 100 строк определяет интерфейс \texttt{AppState} и создаёт store через \texttt{create<AppState>}. Store включает следующие группы данных: навигация (текущая страница, состояние боковой панели), конфигурация теста (режим, сценарий, число виртуальных пользователей, итерации, время прогрева), текущий тест, история тестов, данные реального времени (временные ряды для каждой СУБД), имена и типы подключений, параметры сравнительного анализа.

Все обновления состояния выполняются через диспетчеры (\texttt{setTestConfig}, \texttt{addRealtimeData}, \texttt{setCurrentPage} и др.), что обеспечивает предсказуемость изменений и упрощает отладку.

\subsubsection*{Реализация WebSocket-хука}

Хук \texttt{useTestWebSocket} файла \texttt{hooks/use-test-websocket.ts} объёмом около 290 строк обеспечивает двустороннюю связь с сервером для получения метрик в реальном времени. При монтировании компонента с валидным \texttt{testId} автоматически устанавливается WebSocket-соединение по адресу \texttt{/ws/test/\{testId\}}. Каждые 30 секунд отправляется ping-сообщение для предотвращения разрыва соединения из-за таймаута.

Обработка входящих сообщений реализована через \texttt{handleMessage}:

\begin{figure}
    \begin{minipage}{\linewidth}
\begin{lstlisting}[language=C++]]

const handleMessage = useCallback((event) => {
  const message = JSON.parse(event.data)
  
  switch (message.type) {
    case "metrics":
      const point: TimeSeriesPoint = {
        timestamp: new Date(metricsData.timestamp).getTime(),
        responseTime: metricsData.response_time,

        memoryUsageMB: metricsData.memory_usage_mb,
        cacheHitRatio: metricsData.cache_hit_ratio,

      addRealtimeData(dbKey, point)
      onMetrics?.(metricsData)
      break
    case "status":
      setStatus(statusData.status)
      setProgress(statusData.progress)
      onStatus?.(statusData)
      break
  }
}, [addRealtimeData, onMetrics, onStatus])
\end{lstlisting}

    \label{lst:websocket_handler}
\end{figure}

При разрыве соединения во время выполнения теста производится автоматическое переподключение с интервалом 3 секунды (настраивается через параметр \texttt{reconnectInterval}). Переподключение выполняется только при статусе \texttt{running}, что предотвращает бесконечные попытки подключения после завершения теста. При получении сообщения типа \texttt{metrics} данные преобразуются в \texttt{TimeSeriesPoint} и добавляются в store через \texttt{addRealtimeData}, что вызывает реактивное обновление компонентов дашборда.

\subsubsection*{Визуализация данных}

Визуализация метрик реализована через библиотеку Recharts. На странице дашборда отображаются графики времени отклика, пропускной способности, загрузки CPU и использования RAM в реальном времени. Стилизация интерфейса выполнена через Tailwind CSS v4 с использованием дизайн-токенов (CSS custom properties) для цветов, отступов и размеров шрифтов. Поддерживается тёмная тема, активируемая по умолчанию. Компоненты интерфейса построены на базе библиотеки Radix UI, обеспечивающей доступность (accessibility) и корректную работу с клавиатурой. Уведомления пользователю отображаются через библиотеку Sonner в виде всплывающих сообщений (toast notifications).

\subsection{Контейнеризация и развёртывание}

Развёртывание системы организовано через Docker и Docker Compose. Файл \texttt{docker-compose.yml} определяет сервисы, необходимые для работы системы.

\begin{table}
    \centering
    \caption{Сервисы Docker Compose}
    \label{tab:docker_services}
    \begin{tabularx}{\textwidth}{l l X l}
        \hline
        Сервис & Образ & Назначение & Порт \\
        \hline
        history-db & postgres:15-alpine & База данных истории тестов & 5433:5432 \\
        \hline
    \end{tabularx}
\end{table}

Сервис \texttt{history-db} развёртывается на базе образа \texttt{postgres:15-alpine}. Сервис настроен с именованным томом \texttt{history\_db\_data} для персистентности данных, healthcheck через \texttt{pg\_isready} с интервалом 5 секунд и политикой перезапуска \texttt{unless-stopped}. Порт 5433 проброшен на порт 5432 внутри контейнера для избежания конфликтов с другими экземплярами PostgreSQL. Сервис подключён к пользовательской Docker-сети \texttt{loadtest-network} типа bridge.

Тестовые СУБД развёртываются отдельно через собственные Docker Compose-файлы. MySQL с схемой Sakila развёртывается через контейнер \texttt{sakila} с использованием официального образа MySQL. PostgreSQL с схемой Pagila развёртывается через контейнер \texttt{pagila} с использованием кастомного образа, включающего скрипты инициализации схемы Pagila. Оба контейнера подключаются к той же Docker-сети \texttt{loadtest-network}, что обеспечивает сетевое взаимодействие между компонентами системы.

\subsubsection*{Инициализация и миграции}

Инициализация базы данных истории выполняется через скрипт \texttt{init\_history\_db.py}, создающий все необходимые таблицы согласно моделям SQLAlchemy. Инициализация встроенных сценариев выполняется через скрипт \texttt{init\_scenarios.py}, добавляющий предопределённые сценарии тестирования (\texttt{read\_only}, \texttt{write\_only}, \texttt{mixed\_light}, \texttt{mixed\_heavy}, \texttt{oltp}, \texttt{olap}) с соответствующими SQL-запросами и параметрами.

Для миграций схемы базы данных истории используются скрипты в директории \texttt{backend/scripts/migration/}. Скрипт \texttt{add\_metric\_samples.py} добавляет таблицу \texttt{metric\_samples} для хранения сырых выборок метрик, необходимых для статистического сравнения. Скрипт \texttt{add\_restore\_columns.py} добавляет колонки для управления восстановлением базы данных в таблицу \texttt{test\_runs}. Миграции выполняются вручную перед обновлением системы и не используют автоматический инструмент миграций (Alembic), так как схема базы данных истории изменяется относительно редко.

Конфигурация системы задаётся через файл \texttt{.env}, содержащий переменные окружения: \texttt{HISTORY\_DATABASE\_URL} для подключения к базе данных истории, \texttt{ENCRYPTION\_KEY} для шифрования паролей подключений, \texttt{API\_HOST} и \texttt{API\_PORT} для настройки API-сервера, \texttt{NEXT\_PUBLIC\_WS\_URL} и \texttt{NEXT\_PUBLIC\_API\_URL} для настройки фронтенда. Файл \texttt{.env} не включается в систему контроля версий; вместо него поставляется шаблон \texttt{env.example} с примерами значений.

Процесс первого запуска системы включает следующие шаги: клонирование репозитория, копирование \texttt{env.example} в \texttt{.env} с настройкой параметров, запуск \texttt{docker compose up -d} для развёртывания базы данных истории, выполнение скрипта \texttt{init\_history\_db.py} для создания таблиц, выполнение \texttt{init\_scenarios.py} для загрузки встроенных сценариев, запуск бэкенда через \texttt{uvicorn backend.main:app --reload} и фронтенда через \texttt{npm run dev}.

\subsection{Выводы по главе 3}

В ходе программной реализации системы решены следующие задачи:

\begin{enumerate}[label=\arabic*)]
    \item Обоснован выбор технологического стека: Python/FastAPI для серверной части, Next.js/TypeScript для клиентской части, PostgreSQL для хранения истории, asyncpg и aiomysql для асинхронного доступа к тестируемым СУБД;
    \item Реализован модуль генерации нагрузки (\texttt{LoadTester}) объёмом более 1400 строк с асинхронным выполнением запросов, потоковой отправкой метрик через callback-паттерн, сбором системных и внутренних метрик СУБД, фазой прогрева и механизмом восстановления базы данных;
    \item Реализован callback-механизм (\texttt{TestStreamingCallback}) для трансляции метрик в WebSocket и сохранения в базе данных истории с буферизацией пакетной вставки;
    \item Реализован оркестратор управления состоянием базы данных (\texttt{DatabaseStateManager}) с анализом SQL-запросов, созданием резервных копий через \texttt{SqlBackupStrategy}, восстановлением и верификацией целостности по количеству строк;
    \item Реализован модуль сравнительного анализа (\texttt{ComparisonService}) с автоматическим определением типа сравнения, сбором сырых выборок метрик с fallback-механизмом и применением статистических критериев (Шапиро-Уилка, Уэлча, Манна-Уитни);
    \item Реализован REST API с семью группами эндпоинтов для управления тестами, сценариями, подключениями, историей и настройками; асинхронный запуск тестов через \texttt{BackgroundTasks} с немедленным возвратом \texttt{test\_id};
    \item Реализован слой доступа к данным через паттерн Repository с пакетной вставкой сырых метрик и шифрованием паролей подключений алгоритмом AES-256;
    \item Реализована клиентская часть: Zustand store для управления состоянием, API-клиент с типизированными методами, WebSocket-хук с автоматическим переподключением, семь страниц интерфейса с визуализацией через Recharts;
    \item Реализована контейнеризация: Docker Compose для базы данных истории, отдельные контейнеры для тестовых СУБД (Sakila/MySQL, Pagila/PostgreSQL), скрипты инициализации и миграций.
\end{enumerate}

Результаты реализации представляют собой полностью функциональную систему, готовую к проведению нагрузочного тестирования и сравнительного анализа производительности СУБД. Тестирование системы и анализ результатов рассматриваются в следующей главе.
